### 2. `class`：Java 的业务对象和行为容器

#### 🧩 前端视角类比

Java 的 `class` 和 TS 的 `class` 很像，但 Java 更严格：

- 属性必须声明类型
- 方法必须声明返回值类型
- 文件名通常要和 `public class` 名一致
- Java 更强调封装，字段一般 `private`

#### ☕ Java 实际概念

`class` 是 Java 里最核心的代码组织单位。

一个类通常包含：


字段 field
构造方法 constructor
普通方法 method
getter/setter


#### 📝 代码对比

| TypeScript | Java |
|---|---|
|  class Product { constructor( public id: number, public name: string, public price: number ) {} isExpensive() { return this.price > 100 } }  |  public class Product { private Long id; private String name; private Integer price; public Product(Long id, String name, Integer price) { this.id = id; this.name = name; this.price = price; } public boolean isExpensive() { return this.price > 100; } public Long getId() { return id; } public String getName() { return name; } public Integer getPrice() { return price; } }  |

---

### 3. `record`：Java 里的轻量数据对象

#### 🧩 前端视角类比

`record` 很像 TS 里的只读数据类型：


type User = {
  readonly id: number
  readonly name: string
}


#### ☕ Java 实际概念

`record` 是 Java 里用于承载不可变数据的简洁写法。

它自动生成：

- 构造方法
- 字段
- 访问方法
- `equals`
- `hashCode`
- `toString`

#### 📝 代码对比

| TypeScript | Java |
|---|---|
|  type User = { id: number name: string } function getName(user: User) { return user.name }  |  public record User(Long id, String name) {} public String getName(User user) { return user.name(); }  |

#### 💡 记忆点

- `class`：适合复杂业务对象，可变状态多。
- `record`：适合 DTO、VO、简单数据载体。
- `record` 的字段访问不是 `user.name`，而是 `user.name()`。

### 4. `interface`：Java 的能力契约

#### 🧩 前端视角类比

TS 里的 interface：


interface Repository<T> {
  findById(id: number): T | null
}


Java 里的 interface 也是类似的契约：


public interface Repository<T> {
    T findById(Long id);
}


#### ☕ Java 实际概念

Java 的 `interface` 通常用来定义：

- 某个类必须实现哪些方法
- 分层架构里的抽象能力
- 后续 Spring 里 Service、Mapper 的契约

#### 📝 代码对比

| TypeScript | Java |
|---|---|
|  interface ProductRepo { findById(id: number): Product | null } class MemoryProductRepo implements ProductRepo { findById(id: number) { return null } }  |  public interface ProductRepo { Product findById(Long id); } public class MemoryProductRepo implements ProductRepo { @Override public Product findById(Long id) { return null; } }  |

### 5. `enum`：枚举状态

#### 🧩 前端视角类比

前端常写：


type ProductStatus = 'ON_SALE' | 'OFF_SALE'


或者：


enum ProductStatus {
  ON_SALE = 'ON_SALE',
  OFF_SALE = 'OFF_SALE'
}


Java 中使用：


public enum ProductStatus {
    ON_SALE,
    OFF_SALE
}


#### ☕ Java 实际概念

`enum` 用于定义固定的一组常量，特别适合：

- 商品状态
- 订单状态
- 用户状态
- 支付状态
- 权限类型

#### 📝 代码对比

| TypeScript | Java |
|---|---|
|  type OrderStatus = 'CREATED' | 'PAID' | 'CANCELLED' function canPay(status: OrderStatus) { return status === 'CREATED' }  |  public enum OrderStatus { CREATED, PAID, CANCELLED } public boolean canPay(OrderStatus status) { return status == OrderStatus.CREATED; }  |

### 6. 访问修饰符：`public / private / protected`

#### 🧩 前端视角类比

TS 里你可能这样写：


class UserService {
  private token: string
  public login() {}
}


Java 也是类似：


public class UserService {
    private String token;

    public void login() {}
}


#### ☕ Java 实际概念

| 修饰符 | 含义 | 前端类比 |
|---|---|---|
| `public` | 任何地方都能访问 | export 出去的 API |
| `private` | 只能当前类内部访问 | class 内部私有字段 |
| `protected` | 当前类、子类、同包可访问 | 类继承场景可见 |
| 默认不写 | 同一个 package 可访问 | 同目录模块内部共享 |

#### 💡 记忆点

Java 后端项目里，字段通常写成 `private`，再通过方法访问。

### 7. `static`：属于类，不属于对象

#### 🧩 前端视角类比

TS / JS 里：


class MathUtil {
  static add(a: number, b: number) {
    return a + b
  }
}

MathUtil.add(1, 2)


Java 里一样：


public class MathUtil {
    public static int add(int a, int b) {
        return a + b;
    }
}

MathUtil.add(1, 2);


#### ☕ Java 实际概念

`static` 表示成员属于类本身，而不是某一个实例对象。

常见场景：

- 工具类方法
- 常量
- `main` 方法
- 静态工厂方法

#### 💡 记忆点

`static` 不需要 `new` 对象就能调用。

### 8. `final`：不能再变

#### 🧩 前端视角类比

类似 TS / JS 的 `const`：


const name = 'Java'


Java：


final String name = "Java";


#### ☕ Java 实际概念

`final` 可以修饰：

| 用法 | 含义 |
|---|---|
| `final 变量` | 变量不能重新赋值 |
| `final 方法` | 方法不能被重写 |
| `final 类` | 类不能被继承 |

#### 💡 记忆点

Java 中经常会这样写依赖：


private final ProductMapper productMapper;


表示这个依赖一旦构造时传进来，就不再改变。

这和前端里：


constructor(private readonly productRepo: ProductRepo) {}


很像。

#### 🧩 前端视角类比

| 前端 | Java |
|---|---|
| `Array<T>` | `List<T>` |
| `Map<K, V>` | `Map<K, V>` |
| `Set<T>` | `Set<T>` |

Java 里最常见的集合：

Set<Long> ids = new HashSet<>();

### 10. 泛型：`<T>`

#### 🧩 前端视角类比

TS 泛型：


function success<T>(data: T): ApiResult<T> {
  return { code: 0, data }
}


Java 泛型：


public static <T> ApiResult<T> success(T data) {
    return new ApiResult<>(0, data);
}


#### ☕ Java 实际概念

泛型用于让类、接口、方法支持不同类型，同时保持类型安全。

例如统一返回：


public record ApiResult<T>(Integer code, String message, T data) {
    public static <T> ApiResult<T> success(T data) {
        return new ApiResult<>(0, "success", data);
    }
}


#### 💡 记忆点

- TS 泛型更灵活。
- Java 泛型更严格。
- Java 运行时有类型擦除，Day1 先知道即可，不深挖。

### 11. `Optional`：更显式地处理空值

#### 🧩 前端视角类比

前端常写：


const product = products.find(p => p.id === id)

if (!product) {
  throw new Error('商品不存在')
}


Java 可以写：


Product product = products.stream()
    .filter(p -> p.getId().equals(id))
    .findFirst()
    .orElseThrow(() -> new BizException("商品不存在"));


这里 `findFirst()` 返回的是：


Optional<Product>


#### ☕ Java 实际概念

`Optional<T>` 表示：

> 这个值可能存在，也可能不存在。

常用方法：


optional.isPresent()
optional.orElse(defaultValue)
optional.orElseThrow()
optional.map()


#### 💡 记忆点

`Optional` 不是为了完全消灭 null，而是让“可能为空”这件事更明显。

### 12. Stream API：Java 里的数组链式处理

#### 🧩 前端视角类比

前端：


products
  .filter(p => p.price > 100)
  .map(p => p.name)


Java：


products.stream()
    .filter(p -> p.getPrice() > 100)
    .map(Product::getName)
    .toList();


#### ☕ Java 实际概念

Stream API 是 Java 处理集合的函数式写法，类似 JS 数组的：

- `filter`
- `map`
- `reduce`
- `find`
- `some`
- `every`

#### 📝 代码对比

| TypeScript | Java |
|---|---|
|  const names = products .filter(p => p.price > 100) .map(p => p.name)  |  List<String> names = products.stream() .filter(p -> p.getPrice() > 100) .map(Product::getName) .toList();  |

### 13. 异常处理：`throw / try-catch`

#### 🧩 前端视角类比

前端：


if (!product) {
  throw new Error('商品不存在')
}


Java：


if (product == null) {
    throw new BizException("商品不存在");
}


#### ☕ Java 实际概念

Java 异常分两大类：

| 类型 | 含义 | Day1 建议 |
|---|---|---|
| Checked Exception | 编译器强制你处理 | 先知道 |
| RuntimeException | 运行时异常 | 业务异常优先用它 |

自定义业务异常：


public class BizException extends RuntimeException {
    public BizException(String message) {
        super(message);
    }
}


#### 💡 记忆点

后面 Spring Boot 里会用：


BizException + GlobalExceptionHandler


来做统一错误返回。

## 📝 Demo 3：用 Optional + BizException 实现商品详情查询

### 目标

练习：

- 自定义异常
- `Optional`
- `orElseThrow`
- Service 层思维
- 从 Day2 的分层架构提前过渡

---

### 前端版


type Product = {
  id: number
  name: string
  price: number
}

class BizError extends Error {
  constructor(message: string) {
    super(message)
    this.name = 'BizError'
  }
}

class ProductService {
  constructor(private products: Product[]) {}

  findById(id: number) {
    const product = this.products.find(p => p.id === id)

    if (!product) {
      throw new BizError('商品不存在')
    }

    return product
  }
}

const service = new ProductService([
  { id: 1, name: 'MacBook', price: 12999 }
])

console.log(service.findById(1))
console.log(service.findById(999))


---

### Java 版


package com.example.day01.exception;

public class BizException extends RuntimeException {

    public BizException(String message) {
        super(message);
    }
}



package com.example.day01.service;

import com.example.day01.domain.Product;
import com.example.day01.domain.ProductStatus;
import com.example.day01.exception.BizException;

import java.util.ArrayList;
import java.util.List;

public class ProductService {

    private final List<Product> products = new ArrayList<>();

    public ProductService() {
        products.add(new Product(1L, "MacBook", 12999, 10, ProductStatus.ON_SALE));
        products.add(new Product(2L, "Mouse", 99, 100, ProductStatus.ON_SALE));
    }

    public Product findById(Long id) {
        return products.stream()
                .filter(product -> product.getId().equals(id))
                .findFirst()
                .orElseThrow(() -> new BizException("商品不存在"));
    }
}



package com.example.day01;

import com.example.day01.domain.Product;
import com.example.day01.service.ProductService;

public class App {

    public static void main(String[] args) {
        ProductService productService = new ProductService();

        Product product = productService.findById(1L);
        System.out.println(product.getName());

        Product notFound = productService.findById(999L);
        System.out.println(notFound.getName());
    }
}


---

### 这个 Demo 要掌握什么？

| 知识点 | 解释 |
|---|---|
| `BizException` | 业务异常，后面会接入全局异常处理 |
| `RuntimeException` | 运行时异常，不要求每次都显式 `try-catch` |
| `findFirst()` | 查找第一个匹配结果 |
| `Optional` | 代表可能有值，也可能没值 |
| `orElseThrow()` | 没有值就抛异常 |
| `private final` | 表示这个字段不能被重新赋值 |

function success<T>(data: T): ApiResult<T> {
  return {
    code: 0,
    message: 'success',
    data
  }
}

function fail(message: string): ApiResult<null> {
  return {
    code: 500,
    message,
    data: null
  }
}

const result = success({
  id: 1,
  name: 'MacBook'
})

console.log(result)

## ❓ Day 1 高频疑问点与难点详解

### 疑问 1：Java 的 `class` 和 TS 的 `class` 到底有什么区别？

#### 🧩 前端类比

TS 的类型很多时候是“开发期约束”，运行后很多类型信息就没了。

Java 的类型是语言和运行时都强约束的。

#### ☕ Java 实际解释

TS：


class Product {
  constructor(public id: number) {}
}


Java：


public class Product {
    private Long id;

    public Product(Long id) {
        this.id = id;
    }
}


核心区别：

| 对比点 | TypeScript | Java |
|---|---|---|
| 类型存在阶段 | 主要编译期 | 编译期 + 运行期 |
| 字段声明 | 可以简写 | 通常显式声明 |
| 构造参数属性 | 支持 `public id` 简写 | 不支持这种简写 |
| 文件要求 | 较灵活 | `public class` 通常与文件名一致 |
| 封装习惯 | 可灵活 | 常用 `private + getter/setter` |

#### 💡 记忆点

Java 更像“强约束版 TS class”。

---

### 疑问 2：为什么 Java 字段要写 private，还要 getter？

#### 🧩 前端类比

前端组件里你不会随便让外部直接改内部状态，而是通过 props、emit、action、service 方法来修改。

Java 也是一样：


private Integer stock;


表示库存不能被外部随便改。

#### ☕ Java 实际解释

如果字段全是 public：


product.stock = -999;


业务就失控了。

更好的方式是提供业务方法：


public void decreaseStock(Integer quantity) {
    if (quantity > stock) {
        throw new BizException("库存不足");
    }
    this.stock -= quantity;
}


#### 💡 记忆点

`private` 不是啰嗦，是为了把业务规则收口。

---

### 疑问 3：`Long` 和 `long` 有什么区别？

#### 🧩 前端类比

前端只有 `number`，Java 更细。

Java 分：


基本类型：long
包装类型：Long


#### ☕ Java 实际解释

| 类型 | 含义 |
|---|---|
| `long` | 基本类型，不能为 null |
| `Long` | 对象类型，可以为 null |

数据库 ID 通常用：


private Long id;


因为数据库新增前，ID 可能还没有生成，可以是 `null`。

#### 💡 记忆点

业务对象里的 ID 优先用 `Long`，不是 `long`。

---

### 疑问 4：为什么 ID 要写 `1L`，不能写 `1`？

#### 🧩 前端类比

TS 里你写：


const id: number = 1


就够了。

Java 区分整数类型：


int id = 1;
long id = 1L;


#### ☕ Java 实际解释

`1` 默认是 `int`。

`1L` 表示这是一个 `long` 字面量。

如果构造方法需要：


new Product(Long id, ...)


你应该传：


new Product(1L, ...)


#### 💡 记忆点

看到 Java 里的 ID，先想到 `Long` 和 `1L`。

---

### 疑问 5：为什么 String 比较不能用 `==`？

#### 🧩 前端类比

JS 中：


'a' === 'a'


很自然。

但 Java 的 `==` 对对象比较的是引用地址。

#### ☕ Java 实际解释

错误写法：


if (name == "MacBook") {
}


推荐写法：


if ("MacBook".equals(name)) {
}


对象 ID 比较也推荐：


product.getId().equals(id)


不要写：


product.getId() == id


#### 💡 记忆点

Java 对象比较内容，用 `.equals()`。

---

### 疑问 6：`List<Product>` 为什么不能直接写 `Product[]`？

#### 🧩 前端类比

前端里数组天然好用：


const products: Product[] = []
products.push(product)


Java 数组长度固定，不适合业务列表。

#### ☕ Java 实际解释

Java 数组：


Product[] products = new Product[10];


长度固定。

Java List：


List<Product> products = new ArrayList<>();


可动态添加，业务开发更常用。

#### 💡 记忆点

后端业务列表优先用 `List<T>`，不是数组。

---

### 疑问 7：`ArrayList` 和 `List` 是什么关系？

#### 🧩 前端类比

可以类比为：


interface ProductRepo {}

class MemoryProductRepo implements ProductRepo {}


Java 中：


List<Product> products = new ArrayList<>();


意思是：

> 左边用接口类型接收，右边用具体实现创建。

#### ☕ Java 实际解释

`List` 是接口，定义能力：


add
remove
get
size


`ArrayList` 是具体实现。

这样写更灵活：


List<Product> products = new ArrayList<>();


以后可以换成：


List<Product> products = new LinkedList<>();


调用方不用改。

#### 💡 记忆点

面向接口编程：


List<T> list = new ArrayList<>();


不要上来就写：


ArrayList<T> list = new ArrayList<>();


---

### 疑问 8：Stream API 为什么看起来比 JS 数组麻烦？

#### 🧩 前端类比

JS：


products.filter(p => p.price > 100)


Java：


products.stream()
        .filter(p -> p.getPrice() > 100)
        .toList();


多了两个动作：


stream()
toList()


#### ☕ Java 实际解释

Java 的集合本身不是 Stream。

你需要：

1. `.stream()`：把集合变成流
2. `.filter()` / `.map()`：处理中间过程
3. `.toList()`：收集结果

#### 💡 记忆点

Java Stream 固定三段式：


集合.stream()
    .中间操作()
    .终止操作()

### 疑问 9：为什么 `record` 访问字段是 `user.name()`？

#### 🧩 前端类比

TS：


user.name


Java record：


user.name()


#### ☕ Java 实际解释

`record` 自动生成的是访问方法，不是 public 字段。

例如：


public record User(Long id, String name) {}


等价于自动生成：


public Long id() {
    return id;
}

public String name() {
    return name;
}


#### 💡 记忆点

普通 class 用：


user.getName()


record 用：


user.name()

### 疑问 10：Java 为什么要写 `public static void main(String[] args)`？

#### 🧩 前端类比

Node 项目入口：


node index.js


Java 项目入口：


public static void main(String[] args)


#### ☕ Java 实际解释


public static void main(String[] args) {
}


拆开看：

| 部分 | 含义 |
|---|---|
| `public` | JVM 可以从外部访问 |
| `static` | 不需要创建对象就能调用 |
| `void` | 没有返回值 |
| `main` | JVM 识别的入口方法名 |
| `String[] args` | 命令行参数 |

#### 💡 记忆点

Java 程序入口固定写法，先背下来，后面自然理解。

### 疑问 11：Maven 的 `pom.xml` 和 `package.json` 一样吗？

#### 🧩 前端类比

大体角色类似：


package.json 管 Node 项目
pom.xml 管 Java 项目


#### ☕ Java 实际解释

但是 Maven 更强调生命周期：


mvn clean

mvn package
mvn install


其中 Day1 最重要的是：


mvn clean package


它会：

5. 输出到 `target/`

#### 💡 记忆点

`pom.xml` 不只是依赖文件，也是构建规则文件。

### 疑问 12：为什么 Java 代码必须写 package？

#### 🧩 前端类比

前端用目录和 import 区分模块：


import { ProductService } from './service/ProductService'


Java 用 package 表示命名空间：


package com.example.day01.service;


#### ☕ Java 实际解释

`package` 的作用是：

- 避免类名冲突
- 组织代码层级
- 和目录结构对应

例如：


package com.example.day01.service;


通常对应目录：


src/main/java/com/example/day01/service/


#### 💡 记忆点

Java 的 package 类似前端模块路径，但更正式。
