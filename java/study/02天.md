| 前端 / Node 心智 | Java / Spring Boot 心智 |
|---|---|

| NestJS Controller | Spring Controller |
| service 文件 | `@Service` |
| request body | `@RequestBody` |
| route params | `@PathVariable` |
| query params | `@RequestParam` |
| middleware error handler | `@RestControllerAdvice` |
| `res.json()` | 返回 Java 对象，Spring 自动转 JSON |
| `package.json` | `pom.xml` |
| `.env` / config | `application.yml` |
| dependency injection | 构造器注入 |

# 四、Day 2 必学知识点总览

## 1. `@SpringBootApplication`

### 🧩 前端视角类比

它类似 Next.js / NestJS 项目的启动入口。

前端中你可能有：


const app = express()

app.listen(3000)


Spring Boot 中就是：


SpringApplication.run(Day02Application.class, args);


### ☕ Java 实际概念

`@SpringBootApplication` 是一个组合注解，它大概等于：


@Configuration
@EnableAutoConfiguration
@ComponentScan


你现在先记住：

> 被 `@SpringBootApplication` 标记的类，就是整个 Spring Boot 项目的入口。

### 📝 代码对比

| Express / Node | Spring Boot |
|---|---|
|  import express from 'express' const app = express() app.listen(3000)  |  @SpringBootApplication public class Day02Application { public static void main(String[] args) { SpringApplication.run(Day02Application.class, args); } }  |

---

## 2. `@RestController`

### 🧩 前端视角类比

`@RestController` 就像你在 Next.js 里写的 API route：


export async function GET() {}
export async function POST() {}


或者 NestJS：


@Controller('products')
export class ProductController {}


### ☕ Java 实际概念

`@RestController` 表示这个类专门处理 HTTP 请求，并且方法返回值会自动转成 JSON。

以前传统 Spring MVC 可能要写：


@Controller
@ResponseBody


现在直接写：


@RestController


就行。

---

## 3. `@RequestMapping`

### 🧩 前端视角类比

它类似给路由加统一前缀。

前端 Express：


app.use('/api/products', productRouter)


Java：


@RequestMapping("/api/products")


### ☕ Java 实际概念

`@RequestMapping` 可以放在类上，也可以放在方法上。

放在类上表示统一路径前缀：


@RestController
@RequestMapping("/api/products")
public class ProductController {
}


这个类里的所有接口都会自动带上：


/api/products


---

## 4. `@GetMapping`

### 🧩 前端视角类比

等价于 Express 的：


router.get('/products', handler)


Java 写法：


@GetMapping
public ApiResult<List<ProductVO>> list() {
}


### ☕ Java 实际概念

`@GetMapping` 专门处理 HTTP GET 请求。

常用于：


查询列表
查询详情
搜索

## 5. `@PostMapping`

### 🧩 前端视角类比

等价于：


router.post('/products', handler)

### ☕ Java 实际概念

`@PostMapping` 专门处理 HTTP POST 请求。

常用于：

## 8. `@RequestBody`

### 🧩 前端视角类比

它就是从 `req.body` 里拿 JSON。

Express：


const dto = req.body


Spring Boot：


@RequestBody ProductCreateDTO dto


### ☕ Java 实际概念

当前端发送 JSON：


{
  "name": "MacBook",
  "price": 12999,
  "stock": 10
}


Spring Boot 会自动把 JSON 转成 Java 对象：


ProductCreateDTO

## 9. `@PathVariable`

### 🧩 前端视角类比

它就是从 URL 路径参数里取值。

Express：


const id = req.params.id


Spring Boot：


@PathVariable Long id


### ☕ Java 实际概念

对于接口：


GET /api/products/1


Java 方法：


@GetMapping("/{id}")
public ApiResult<ProductVO> detail(@PathVariable Long id) {
}


`id` 就会拿到 `1`。

## 10. `@RequestParam`

### 🧩 前端视角类比

它就是从 query string 里取值。

前端请求：


GET /api/products?keyword=Mac&minPrice=1000


Express：


const keyword = req.query.keyword


Spring Boot：


@RequestParam(required = false) String keyword


### ☕ Java 实际概念

`@RequestParam` 用来接收查询参数。


@GetMapping
public ApiResult<List<ProductVO>> list(
        @RequestParam(required = false) String keyword,
        @RequestParam(required = false) Integer minPrice,
        @RequestParam(required = false) Integer maxPrice
) {
}

## 11. `@Service`

### 🧩 前端视角类比

`@Service` 就像你前端项目里的 `services/productService.ts`。

例如：


export const productService = {
  list() {},
  create(dto) {}
}


Java 中：


@Service
public class ProductService {
}


### ☕ Java 实际概念

`@Service` 表示这是业务逻辑层。

Controller 不应该写太多业务逻辑，只负责：


接请求
取参数
调用 Service
返回结果


真正的业务判断，比如：


商品是否存在
价格是否合法
库存是否足够
新增商品
删除商品


应该放在 Service。

## 12. 构造器注入

### 🧩 前端视角类比

类似 NestJS：


constructor(private readonly productService: ProductService) {}


Java：


private final ProductService productService;

public ProductController(ProductService productService) {
    this.productService = productService;
}


### ☕ Java 实际概念

Spring 会自动帮你创建 `ProductService` 对象，然后传给 `ProductController`。

这就是依赖注入，也叫 DI。

推荐写法是：


private final ProductService productService;

public ProductController(ProductService productService) {
    this.productService = productService;
}


不推荐 Day 2 一开始就这样写：


@Autowired
private ProductService productService;


因为构造器注入更清晰、更安全、更容易测试。

## 14. 统一返回 `ApiResult<T>`

### 🧩 前端视角类比

前端通常希望后端返回统一结构：


type ApiResult<T> = {
  code: number
  message: string
  data: T
}


Java 中一样：


public record ApiResult<T>(
        Integer code,
        String message,
        T data
) {}


### ☕ Java 实际概念

不要让每个接口随便返回不同格式。

推荐统一返回：


{
  "code": 0,
  "message": "success",
  "data": {}
}


失败时：


{
  "code": 500,
  "message": "商品不存在",
  "data": null
}

### 🧩 前端视角类比

app.use((err, req, res, next) => {
  res.json({
    code: 500,
    message: err.message

}

## `ApiResult.java`

### 🧩 前端版


type ApiResult<T> = {
  code: number
  message: string
  data: T | null
}

function success<T>(data: T): ApiResult<T> {
  return {
    code: 0,
    message: 'success',
    data
  }
}

function fail(message: string): ApiResult<null> {
  return {
    code: 500,
    message,
    data: null
  }
}


### ☕ Java 版


package com.example.day02.common;

public record ApiResult<T>(
        Integer code,
        String message,
        T data
) {

    public static <T> ApiResult<T> success(T data) {
        return new ApiResult<>(0, "success", data);
    }

    public static ApiResult<Void> success() {
        return new ApiResult<>(0, "success", null);
    }

    public static ApiResult<Void> fail(String message) {
        return new ApiResult<>(500, message, null);
    }

    public static ApiResult<Void> fail(Integer code, String message) {
        return new ApiResult<>(code, message, null);
    }
}


### 💡 关键记忆点

- `ApiResult<T>` 类似 TS 的泛型类型。
- `record` 适合做只读数据结构。
- `success(T data)` 里的 `<T>` 是 Java 方法泛型声明。
- `Void` 表示没有 data。

## 2. `GlobalExceptionHandler.java`

### 🧩 前端版


app.use((err, req, res, next) => {
  if (err instanceof BizError) {
    return res.json({
      code: err.code,
      message: err.message,
      data: null
    })
  }

  res.json({
    code: 500,
    message: '系统异常',

### ☕ Java 版


package com.example.day02.exception;

import com.example.day02.common.ApiResult;
import org.springframework.web.bind.MethodArgumentNotValidException;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.RestControllerAdvice;

@RestControllerAdvice
public class GlobalExceptionHandler {

    /**
     * 处理业务异常
     */
    @ExceptionHandler(BizException.class)
    public ApiResult<Void> handleBizException(BizException e) {
        return ApiResult.fail(e.getCode(), e.getMessage());
    }

    /**
     * 处理参数校验异常
     */
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ApiResult<Void> handleValidationException(MethodArgumentNotValidException e) {
        String message = e.getBindingResult()
                .getFieldErrors()
                .stream()
                .findFirst()
                .map(error -> error.getField() + " " + error.getDefaultMessage())
                .orElse("参数校验失败");

        return ApiResult.fail(400, message);
    }

    /**
     * 兜底异常
     */
    @ExceptionHandler(Exception.class)
    public ApiResult<Void> handleException(Exception e) {
        e.printStackTrace();
        return ApiResult.fail(500, "系统异常：" + e.getMessage());
    }
}


### 💡 关键记忆点

- Controller 里不要到处 `try-catch`。
- 业务错误直接 `throw new BizException("商品不存在")`。
- 全局异常处理器统一把异常转成 JSON。

## 2. `ProductUpdateDTO.java`


package com.example.day02.dto;

import com.example.day02.entity.ProductStatus;

import jakarta.validation.constraints.NotNull;

public class ProductUpdateDTO {

    @NotBlank(message = "商品名称不能为空")
    private String name;

    @NotNull(message = "商品价格不能为空")
    @Min(value = 1, message = "商品价格必须大于 0")

## `ProductService.java`

这里先用内存集合模拟数据库。


package com.example.day02.service;

import com.example.day02.dto.ProductCreateDTO;
import com.example.day02.dto.ProductUpdateDTO;
import com.example.day02.entity.Product;
import com.example.day02.entity.ProductStatus;
import com.example.day02.exception.BizException;
import com.example.day02.vo.ProductVO;
import org.springframework.stereotype.Service;

import java.util.List;
import java.util.concurrent.CopyOnWriteArrayList;
import java.util.concurrent.atomic.AtomicLong;

@Service
public class ProductService {

    /**
     * Day2 暂时用内存 List 模拟数据库。
     *
     * 注意：
     * 普通 ArrayList 线程不安全。
     * Web 请求是多线程的，所以这里用 CopyOnWriteArrayList 先降低并发问题。
     *
     * Day3 接入 MySQL 后，这里会换成 ProductMapper。
     */
    private final List<Product> products = new CopyOnWriteArrayList<>();

    /**
     * 模拟数据库自增 ID。
     */
    private final AtomicLong idGenerator = new AtomicLong(100);

    public ProductService() {
        products.add(new Product(1L, "MacBook Pro", 12999, 10, ProductStatus.ON_SALE));
        products.add(new Product(2L, "Magic Mouse", 599, 100, ProductStatus.ON_SALE));
        products.add(new Product(3L, "Mechanical Keyboard", 899, 30, ProductStatus.ON_SALE));
        products.add(new Product(4L, "Studio Display", 11999, 5, ProductStatus.OFF_SALE));
        products.add(new Product(5L, "USB Cable", 99, 0, ProductStatus.ON_SALE));
    }

    /**
     * 查询商品列表，支持 keyword、minPrice、maxPrice 过滤。
     */
    public List<ProductVO> list(String keyword, Integer minPrice, Integer maxPrice) {
        return products.stream()
                .filter(product -> {
                    if (keyword == null || keyword.isBlank()) {
                        return true;
                    }
                    return product.getName().toLowerCase().contains(keyword.toLowerCase());
                })
                .filter(product -> {
                    if (minPrice == null) {
                        return true;
                    }
                    return product.getPrice() >= minPrice;
                })
                .filter(product -> {
                    if (maxPrice == null) {
                        return true;
                    }
                    return product.getPrice() <= maxPrice;
                })
                .map(ProductVO::from)
                .toList();
    }

    /**
     * 查询商品详情。
     */
    public ProductVO detail(Long id) {
        Product product = findProductById(id);
        return ProductVO.from(product);
    }

    /**
     * 新增商品。
     */
    public ProductVO create(ProductCreateDTO dto) {
        boolean nameExists = products.stream()
                .anyMatch(product -> product.getName().equalsIgnoreCase(dto.getName()));

        if (nameExists) {
            throw new BizException(400, "商品名称已存在");
        }

        Long id = idGenerator.incrementAndGet();

        Product product = new Product(
                id,
                dto.getName(),
                dto.getPrice(),
                dto.getStock(),
                ProductStatus.ON_SALE
        );

        products.add(product);

        return ProductVO.from(product);
    }

    /**
     * 更新商品。
     */
    public ProductVO update(Long id, ProductUpdateDTO dto) {
        Product product = findProductById(id);

        boolean nameExists = products.stream()
                .anyMatch(item ->
                        !item.getId().equals(id)
                                && item.getName().equalsIgnoreCase(dto.getName())
                );

        if (nameExists) {
            throw new BizException(400, "商品名称已存在");
        }

        product.setName(dto.getName());
        product.setPrice(dto.getPrice());
        product.setStock(dto.getStock());
        product.setStatus(dto.getStatus());

        return ProductVO.from(product);
    }

    /**
     * 删除商品。
     */
    public void delete(Long id) {
        Product product = findProductById(id);
        products.remove(product);
    }

    /**
     * 商品上架。
     */
    public ProductVO onSale(Long id) {
        Product product = findProductById(id);
        product.setStatus(ProductStatus.ON_SALE);
        return ProductVO.from(product);
    }

    /**
     * 商品下架。
     */
    public ProductVO offSale(Long id) {
        Product product = findProductById(id);
        product.setStatus(ProductStatus.OFF_SALE);
        return ProductVO.from(product);
    }

    /**
     * 公共方法：根据 ID 找商品。
     */
    private Product findProductById(Long id) {
        return products.stream()
                .filter(product -> product.getId().equals(id))
                .findFirst()
                .orElseThrow(() -> new BizException(404, "商品不存在，id=" + id));
    }
}

# 十六、学习中高频疑问点与难点详解

---

## 疑问 1：为什么 Controller 不直接写业务逻辑？

### 🧩 前端视角类比

前端里你一般不会把所有请求逻辑、状态处理、数据转换都塞进组件里。

不推荐：


function ProductPage() {
  // 里面写请求、过滤、转换、异常、状态更新全部逻辑
}


更推荐：


components/ProductPage.tsx
services/productService.ts
api/productApi.ts
types/product.ts


Java 也是一样。

### ☕ Java 实际解释

Controller 应该只做三件事：


1. 接收请求
2. 调用 Service
3. 返回结果


业务判断放在 Service。

错误写法：


@PostMapping
public ApiResult<ProductVO> create(@RequestBody ProductCreateDTO dto) {
    if (dto.getPrice() <= 0) {
        throw new BizException("价格错误");
    }

    Product product = new Product();
    product.setName(dto.getName());
    product.setPrice(dto.getPrice());

    products.add(product);

    return ApiResult.success(ProductVO.from(product));
}


更好的写法：


@PostMapping
public ApiResult<ProductVO> create(@Valid @RequestBody ProductCreateDTO dto) {
    return ApiResult.success(productService.create(dto));
}


### 💡 记忆点

Controller 越薄越好，Service 才是业务核心。

---

## 疑问 2：DTO、VO、Entity 到底为什么要分开？

### 🧩 前端视角类比

前端也会区分：


type LoginForm = {
  username: string
  password: string
}

type UserInfo = {
  id: number
  username: string
  avatar: string
}


你不会把密码返回给页面。

Java 后端更强调这种分层。

### ☕ Java 实际解释

| 类型 | 作用 | 类比 |
|---|---|---|
| DTO | 前端传给后端的数据 | request payload |
| Entity | 后端内部业务对象/数据库对象 | database model |
| VO | 后端返回给前端的数据 | response view model |

例如：


public class ProductCreateDTO {
    private String name;
    private Integer price;
    private Integer stock;
}



public class Product {
    private Long id;
    private String name;
    private Integer price;
    private Integer stock;
    private ProductStatus status;
}



public record ProductVO(
        Long id,
        String name,
        Integer price,
        Boolean available
) {}


### 💡 记忆点

不要偷懒一个类走天下。

真实项目里：


DTO 负责接收请求
Entity 负责内部数据
VO 负责响应前端


---

## 疑问 3：为什么要统一返回 ApiResult？

### 🧩 前端视角类比

如果后端接口有的返回：


[]


有的返回：


{
  "list": []
}


有的失败时返回：


"error"


前端会很痛苦。

你希望所有接口都长这样：


type ApiResult<T> = {
  code: number
  message: string
  data: T
}


### ☕ Java 实际解释

统一返回后，前端可以统一封装 axios：


request.interceptors.response.use((response) => {
  const result = response.data

  if (result.code !== 0) {
    throw new Error(result.message)
  }

  return result.data
})


Java 后端统一：


return ApiResult.success(data);


失败：


return ApiResult.fail("商品不存在");


### 💡 记忆点

统一返回是前后端协作的契约。

---

## 疑问 4：为什么异常要全局处理？

### 🧩 前端视角类比

Express 里你不会每个接口都这样：


router.get('/products/:id', async (req, res) => {
  try {
    // ...
  } catch (e) {
    res.json({ code: 500, message: e.message })
  }
})


而是统一错误中间件。

Spring Boot 也是一样。

### ☕ Java 实际解释

有了全局异常处理后，Service 可以专注业务：


if (product == null) {
    throw new BizException(404, "商品不存在");
}


异常会自动被：


@RestControllerAdvice


捕获并转换成 JSON。

### 💡 记忆点

Service 抛异常，GlobalExceptionHandler 统一兜底。

---

## 疑问 5：`@RequestBody` 和 `@RequestParam` 有什么区别？

### 🧩 前端视角类比

| 前端位置 | Java 注解 |
|---|---|
| `req.body` | `@RequestBody` |
| `req.query` | `@RequestParam` |
| `req.params` | `@PathVariable` |

### ☕ Java 实际解释

请求：


POST /api/products
Content-Type: application/json

{
  "name": "MacBook",
  "price": 12999
}


对应：


@PostMapping
public ApiResult<ProductVO> create(@RequestBody ProductCreateDTO dto) {
}


请求：


GET /api/products?keyword=Mac


对应：


@GetMapping
public ApiResult<List<ProductVO>> list(@RequestParam String keyword) {
}


请求：


GET /api/products/1


对应：


@GetMapping("/{id}")
public ApiResult<ProductVO> detail(@PathVariable Long id) {
}


### 💡 记忆点


body    -> @RequestBody
query   -> @RequestParam
path    -> @PathVariable

## 疑问 6：为什么 Service 上加 `@Service`，Controller 上加 `@RestController`？

### 🧩 前端视角类比

NestJS 里你会写：


@Injectable()
export class ProductService {}



@Controller('products')
export class ProductController {}


Spring Boot 类似。

### ☕ Java 实际解释

Spring Boot 启动时会扫描这些注解：


@RestController
@Service
@Component
@Repository


然后自动创建对象，放到 Spring 容器里。

你可以理解为 Spring 内部维护了一个对象池：


ProductController 实例
ProductService 实例
GlobalExceptionHandler 实例


然后自动帮你组装依赖。

### 💡 记忆点

加了 Spring 注解，这个类才会被 Spring 管理。

---

## 疑问 7：构造器注入为什么比 `new ProductService()` 好？

### 🧩 前端视角类比

如果你在组件里到处手动 new：


const service = new ProductService()


测试、替换、mock 都会很麻烦。

NestJS / Angular 会用 DI 容器帮你注入。

### ☕ Java 实际解释

错误写法：


@RestController
public class ProductController {

    private ProductService productService = new ProductService();
}


问题：


1. ProductService 不受 Spring 管理
2. 以后 ProductService 依赖 Mapper 时会失效
3. 不方便测试
4. 不方便替换实现


推荐写法：


private final ProductService productService;

public ProductController(ProductService productService) {
    this.productService = productService;
}


### 💡 记忆点

在 Spring 项目里，不要自己随便 `new Service`。

## 疑问 8：为什么 Java 返回对象会自动变成 JSON？

### 🧩 前端视角类比

Express 里你写：


res.json(product)


Spring Boot 中你直接：


return ApiResult.success(product);


### ☕ Java 实际解释

因为 `spring-boot-starter-web` 默认集成了 Jackson。

Jackson 会自动把 Java 对象序列化成 JSON。

例如 Java 对象：


new ProductVO(1L, "MacBook", 12999, 10, ProductStatus.ON_SALE, true)


返回给前端会变成：


{
  "id": 1,
  "name": "MacBook",
  "price": 12999,
  "stock": 10,
  "status": "ON_SALE",
  "available": true
}


### 💡 记忆点

Spring Boot 默认帮你做 JSON 转换，不需要手写 `JSON.stringify`。

## 疑问 9：为什么 `record` 适合 VO，不一定适合 Entity？

### 🧩 前端视角类比

VO 更像只读 response：


type ProductVO = Readonly<{
  id: number
  name: string
}>


Entity 常常需要修改：


product.stock = product.stock - 1


### ☕ Java 实际解释

`record` 默认字段不可变，适合：


DTO
VO
简单返回对象
配置对象


普通 `class` 适合：


Entity
需要修改字段的对象
有较多业务行为的对象


Day 2 里：


public record ProductVO(...) {}


比较合适。

而：


public class Product {}


比较合适。

### 💡 记忆点

只读数据用 record，需要变更状态用 class。

## 疑问 10：为什么用 `CopyOnWriteArrayList` 而不是 `ArrayList`？

### 🧩 前端视角类比

Node.js 大部分时候是单线程事件循环，你很少直接考虑多个线程同时修改数组。

但 Java Web 服务默认是多线程处理请求。

### ☕ Java 实际解释

如果多个请求同时新增、删除商品，普通 `ArrayList` 可能出现并发问题。

Day 2 还没学数据库，临时用：


private final List<Product> products = new CopyOnWriteArrayList<>();


它比 `ArrayList` 更适合简单并发读写。

不过真实项目不会用内存 List 存业务数据，Day 3 会换成 MySQL。

### 💡 记忆点

Day 2 内存 List 只是模拟数据库，不是生产方案。
