### 🔨 Step 3：写Repository——内存数据库（1h）

**作业内容**：模拟数据库操作，用 `ConcurrentHashMap` 做存储

```java
// repository/UserRepository.java
package com.myapp.repository;

import com.myapp.entity.User;
import java.util.*;
import java.util.concurrent.ConcurrentHashMap;
import java.util.concurrent.atomic.AtomicLong;

public class UserRepository {
    private final Map<Long, User> store = new ConcurrentHashMap<>();
    private final AtomicLong idGenerator = new AtomicLong(1);

    public User save(User user) {
        Long id = idGenerator.getAndIncrement();
        User saved = new User(id, user.username(), user.email(), user.age());
        store.put(id, saved);
        return saved;
    }

    public Optional<User> findById(Long id) {
        return Optional.ofNullable(store.get(id));
    }

    public List<User> findAll() {
        return new ArrayList<>(store.values());
    }

    public boolean deleteById(Long id) {
        return store.remove(id) != null;
    }

    public List<User> findByAgeGreaterThan(int age) {
        return store.values().stream()
            .filter(u -> u.age() > age)
            .toList();
    }
}
```

### 🔨 Step 4：写Service——业务逻辑 + 异常处理（1.5h）

**作业内容**：

```java
// service/UserService.java
package com.myapp.service;

import com.myapp.entity.User;
import com.myapp.repository.UserRepository;
import com.myapp.dto.*;
import java.util.List;

public class UserService {
    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public UserResponse createUser(CreateUserRequest request) {
        // 手写参数校验（体会没有 @Valid 注解的痛！）
        if (request.username() == null || request.username().isBlank()) {
            throw new IllegalArgumentException("用户名不能为空");
        }
        if (request.age() < 0 || request.age() > 150) {
            throw new IllegalArgumentException("年龄不合法");
        }

        User user = new User(null, request.username(), request.email(), request.age());
        User saved = userRepository.save(user);
        return toResponse(saved);
    }

    public UserResponse getUserById(Long id) {
        return userRepository.findById(id)
            .map(this::toResponse)
            .orElseThrow(() -> new RuntimeException("用户不存在: " + id));
    }

    public List<UserResponse> getAllUsers() {
        return userRepository.findAll().stream()
            .map(this::toResponse)
            .toList();
    }

    public List<UserResponse> getAdults() {
        return userRepository.findByAgeGreaterThan(17).stream()
            .map(this::toResponse)
            .toList();
    }

    private UserResponse toResponse(User user) {
        return new UserResponse(user.id(), user.username(), user.email(), user.age());
    }
}
```

### 🔨 Step 5：写Main入口——控制台交互（2h）——**最难的一步！**

**作业内容**：手动依赖注入 + 控制台菜单 + switch路由分发

```java
// Main.java
package com.myapp;

import com.myapp.repository.UserRepository;
import com.myapp.service.UserService;
import com.myapp.dto.CreateUserRequest;
import java.util.Scanner;

public class Main {
    public static void main(String[] args) {
        // ⚡ 手动依赖注入——现在你亲手做，下周Spring Boot @Autowired 帮你做
        UserRepository userRepo = new UserRepository();
        UserService userService = new UserService(userRepo);

        Scanner scanner = new Scanner(System.in);

        while (true) {
            System.out.println("\n=== 用户管理系统 ===");
            System.out.println("1. 创建用户");
            System.out.println("2. 查看所有用户");
            System.out.println("3. 查找用户");
            System.out.println("4. 查看成年用户（Stream API）");
            System.out.println("5. 退出");
            System.out.print("请选择: ");

            int choice = scanner.nextInt();
            scanner.nextLine(); // ⚠️ 消耗换行符——前端不需要管这个！

            try {
                switch (choice) {
                    case 1 -> {
                        System.out.print("用户名: ");
                        String name = scanner.nextLine();
                        System.out.print("邮箱: ");
                        String email = scanner.nextLine();
                        System.out.print("年龄: ");
                        int age = scanner.nextInt();

                        var request = new CreateUserRequest(name, email, age);
                        var response = userService.createUser(request);
                        System.out.println("✅ 创建成功: " + response);
                    }
                    case 2 -> {
                        var users = userService.getAllUsers();
                        if (users.isEmpty()) {
                            System.out.println("暂无用户");
                        } else {
                            users.forEach(System.out::println);
                        }
                    }
                    case 3 -> {
                        System.out.print("输入用户ID: ");
                        Long id = scanner.nextLong();
                        var user = userService.getUserById(id);
                        System.out.println(user);
                    }
                    case 4 -> {
                        var adults = userService.getAdults();
                        System.out.println("成年用户数: " + adults.size());
                        adults.forEach(System.out::println);
                    }
                    case 5 -> {
                        System.out.println("再见！");
                        return;
                    }
                }
            } catch (Exception e) {
                System.err.println("❌ 错误: " + e.getMessage());
            }
        }
    }
}
```

## 🔴 难点1：包（package）和目录结构——"为什么Java要有包？"

### 前端类比

```typescript
// 前端你直接用 import 路径
import { User } from '../models/user';
import { UserService } from '../services/userService';
```

```java
// Java 必须用 package 声明 + import
package com.myapp.service;           // 声明我在哪个包

import com.myapp.entity.User;        // 引入其他包的类
import com.myapp.repository.UserRepository;
```

### 为什么设计成这样？

| 维度 | 前端（TS/JS） | Java |
|------|-------------|------|
| **文件定位** | 相对路径 `../models/user` | 全限定类名 `com.myapp.entity.User` |
| **冲突解决** | 不同目录同名文件没事 | 不同包同名类没事，但包名必须不同 |
| **编译要求** | 不需要，运行时解析 | 必须和目录结构完全匹配 |

### 🟡 避坑提醒
```bash
# ❌ 错误：Java文件在 src/com/myapp/entity/User.java
# 但文件里写的 package com.other;
# 编译报错！

# ✅ 必须保持一致
# 目录：src/com/myapp/entity/User.java
# 代码：package com.myapp.entity;
```

## 🔴 难点2：手动依赖注入——"为什么要 new 来 new 去？"

### 前端类比

```typescript
// 前端你一般这样写（直接 import）
import { userApi } from './services/userApi';

function UserList() {
  const [users, setUsers] = useState([]);

  useEffect(() => {
    userApi.fetchUsers().then(setUsers);  // 直接调
  }, []);
}
```

```java
// Java 你必须手动"注入"
public class Main {
    public static void main(String[] args) {
        UserRepository userRepo = new UserRepository();    // 先创建仓库
        UserService userService = new UserService(userRepo); // 把仓库"注入"到服务

        // 然后才能用
        userService.getAllUsers();
    }
}
```

### 为什么设计成这样？

- **前端**：模块系统是静态的，`import` 只是引入代码
- **Java**：面向对象要求——**一个类如果要依赖另一个类，必须显式拿到它的实例**
- 这就是 **依赖注入（DI）** 的雏形。Spring Boot 的 IOC 容器就是帮你自动做这个"new 来 new 去"的过程

### 🟡 避坑提醒
```java
// ❌ 错误——在 Service 里 new Repository
public class UserService {
    private UserRepository userRepository = new UserRepository(); // 紧耦合！
}

// ✅ 正确——通过构造器传入（松耦合）
public class UserService {
    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
}
```

> **为什么？** 如果 `new` 在 Service 里，将来你想换数据库实现（比如改为MySQL），就得改 Service 代码。通过构造器传入，你只需要换传入的对象。

## 🔴 难点3：`public static void main(String[] args)`——"为什么这么长？"

### 前端类比

```typescript
// 前端入口超简单
// index.ts
console.log("Hello");
// 直接运行
```

```java
// Java 入口——必须这样写
public class Main {
    public static void main(String[] args) {
        System.out.println("Hello");
    }
}
```

### 逐词解释（前端视角）

| 部分 | 含义 | 前端类比 |
|------|------|----------|
| `public` | JVM可以从任何地方访问它 | ≈ `export` |
| `static` | 不需要 new Main() 就能运行 | ≈ 类上的静态方法 |
| `void` | 不返回任何值 | ≈ `function main(): void` |
| `main` | 方法名，JVM约定找这个名字 | ≈ 入口文件约定叫 `index.ts` |
| `String[] args` | 命令行参数 | ≈ `process.argv` |

### 🟡 避坑提醒——最常见的3个编译错误

```java
// ❌ 坑1：方法名写错
public static void mian(String[] args) { }  // main 拼成 mian

// ❌ 坑2：少了 static
public void main(String[] args) { }  // 没有static，JVM不认这是入口

// ❌ 坑3：String 小写了
public static void main(string[] args) { }  // 应该大写 String！
```

## 🔴 难点4：`Scanner` 和 `nextInt()`/`nextLine()` 的换行问题

### 前端类比

```typescript
// 前端从控制台读输入
const readline = require('readline');
// 或者 prompt() 在浏览器里
const name = prompt("请输入名字:");
```

```java
// Java——用 Scanner
Scanner scanner = new Scanner(System.in);
int age = scanner.nextInt();   // 只读了数字
scanner.nextLine();            // ⚠️ 必须消耗掉换行符！
String name = scanner.nextLine(); // 才能读到字符串
```

### 为什么会有这个问题？

- `nextInt()` 只读取数字，不读取数字后面的**换行符（\n）**
- 紧接着的 `nextLine()` 会直接读到那个残留的换行符，**跳过了用户输入**
- 解决方法：在 `nextInt()` 后面加一个 `scanner.nextLine()` 消耗换行符

### 前端类比

```javascript
// 前端也有类似问题
const age = parseInt(prompt("年龄:"));
// 但 prompt() 是整行读，不存在残留问题
// Java 的问题是"部分读取"导致的
```

## 🔴 难点5：DTO（数据传输对象）——"为什么要有这么多没用的类？"

### 前端类比

```typescript
// 前端——随手一个对象搞定
const user = {
  id: 1,
  name: '小明',
  email: 'xiao@ming.com',
  password: '123456',      // ⚠️ 不应该给前端！
  createdAt: '2024-01-01'  // 前端不需要这个
};
```

```java
// Java——必须拆成3个
// 1. 用户请求（前端传过来的）
public record CreateUserRequest(String username, String email, Integer age) { }

// 2. 用户实体（数据库里存的）
public record User(Long id, String username, String email, Integer age, String password, LocalDateTime createdAt) { }

// 3. 用户响应（返回给前端的）
public record UserResponse(Long id, String username, String email, Integer age) { }
```

### 为什么设计成这样？

| 前端 | Java |
|------|------|
| 一个JSON搞定一切 | 每层各管各的数据 |
| `const obj = { a: 1, b: 2 }` | `Req → Entity → Resp` 三层转换 |
| 灵活但混乱 | 啰嗦但安全 |

> **核心思想**：不同层关心的数据不一样。前端只需要 `{id, name}`，但数据库有 `{password, createdAt}`。DTO 就是"我只给你看你该看的数据"。

### 🟡 避坑提醒
```java
// 不要直接把 Entity 返回给前端！
public User getUser(Long id) {
    return userRepository.findById(id).get();  // ❌ 会把 password 也返回
}

// 要转成 Response
public UserResponse getUser(Long id) {
    return userRepository.findById(id)
        .map(u -> new UserResponse(u.id(), u.username(), u.email(), u.age()))
        .orElseThrow(() -> new RuntimeException("用户不存在"));
}
```

## 🔴 难点6：编译和运行的命令——"javac 和 java 的区别"

### 前端类比

```bash

```

```bash
# Java
javac Main.java → Main.class  # 编译（.class是字节码）

```

### 🟡 最常翻车的坑

```bash

java src/com/myapp/Main.java  # ❌ 大错特错
```

## 🟢 Java 让你爽到的地方

| # | 爽点 | 前端对照 |
|---|------|---------|

# ❓ Week 1 学习中最高频的疑问与难点详解

## ❓ 疑问1：「没有对象字面量」——为什么不能 `{ name: '小明' }`？

### 前端习惯
```javascript
const user = { name: '小明', age: 18 };
// 随手一个对象
```

### Java 为什么不行？
```java
// ❌ Java没有对象字面量
var user = { name: "小明", age: 18 };  // 编译报错！

// ✅ 必须这样
public class User {
    private String name;
    private int age;
    public User(String name, int age) {
        this.name = name;
        this.age = age;
    }
}
User user = new User("小明", 18);
```

**原因**：Java是**强类型编译型语言**。编译器在编译时就要知道对象的"形状"（有哪些字段、什么类型）。对象字面量是运行时的东西，编译器没法检查。而TS有类型推导，所以可以。

**但有了 Record 后**：
```java
public record User(String name, int age) {}
// 用起来也还算简洁
var user = new User("小明", 18);
```

## ❓ 疑问2：「`.size()` / `.length` / `.length()` 到底用哪个？」

### 速记口诀

| 场景 | 写法 | 原因 |
|------|------|------|
| 数组的长度 | **`arr.length`**（属性，不是方法） | 数组是特殊对象，length是属性 |
| 字符串的长度 | **`str.length()`**（方法，要加括号） | String是类，length()是方法 |
| 集合的大小 | **`list.size()`**（方法） | Collection接口的方法 |

### 前端对比
```javascript
const arr = [1, 2, 3];
arr.length;          // 属性
"hello".length;      // 属性——和Java不一样！
```

**记忆技巧**：
- 只有**数组**是 `.length`（没括号）
- **其他所有**都是有括号的方法 `.length()` 或 `.size()`

## ❓ 疑问3：「检查型异常（Checked Exception）——为什么必须catch？」

### 前端类比

```javascript
// 前端——不处理也没事（程序直接挂）
const data = JSON.parse(invalidJson); // 报错就报错了
console.log("继续运行"); // 不会执行到这里
```

```java
// Java——不处理就不让你编译！
public static void main(String[] args) {
    Thread.sleep(1000); // ❌ 编译报错！
    // Unhandled exception: InterruptedException
}
// 必须这样
public static void main(String[] args) throws InterruptedException {
    Thread.sleep(1000); // ✅ 要么 throws，要么 try-catch
}
```

### 为什么Java这么严格？

前端哲学：**"你报错关我什么事，运行时自己处理"**
Java哲学：**"编译器替你把所有可能出问题的地方标出来，让你提前处理"**

**类比**：就像VS Code里红色波浪线——Java编译器在编译时就帮你画出来了，不修好就别想运行。

## ❓ 疑问4：「类型擦除（Type Erasure）——为什么 `List<String>` 运行时变成 `List`？」

### 前端类比

```typescript
// TS——类型信息保留到运行时？
function getType<T>(arg: T): string {
    return typeof arg; // 但 TS 编译后类型信息也没了
}
```

```java
// Java——泛型信息在运行时被擦除
List<String> strings = new ArrayList<>();
List<Integer> integers = new ArrayList<>();

strings.getClass() == integers.getClass(); // true！
// 运行时都是 ArrayList，不区分 String 和 Integer
```

### 为什么设计成这样？

**Java 1.0 就有 `List`（没有泛型），Java 5.0 才加泛型。为了向后兼容，编译后擦除泛型信息。**

**实际影响**：
```java
// ❌ 不能这样写
if (obj instanceof List<String>) { } // 编译报错！

// ✅ 只能这样
if (obj instanceof List) { }  // 可以
```

**前端对照**：就像ES5没有class，ES6加了class，但编译回ES5后class就变成了function。

## ❓ 疑问5：「`static` 到底是个啥？什么时候用？」

### 前端类比

```javascript
// 前端——没有 static 的概念，只有"导出一个变量"
export const BASE_URL = "http://api.example.com";

export function formatDate(date) {
    return date.toISOString();
}
```

```java
// Java——static 表示"属于类，不属于实例"
public class Constants {
    public static final String BASE_URL = "http://api.example.com";

    public static String formatDate(Date date) {
        return date.toString();
    }
}

// 调用方式
Constants.BASE_URL;          // 直接通过类名访问
Constants.formatDate(now);   // 不需要 new Constants()
```

### 记忆口诀

| `static` 含义 | 前端对照 | 例子 |
|--------------|---------|------|
| **属于类，所有实例共享** | ≈ 模块导出的全局变量 | `static int count` |
| **不需要new就能用** | ≈ `Math.max()`（静态方法） | `Integer.parseInt()` |
| **不能访问非静态成员** | ≈ 纯函数不能访问this | `static void main()` |

## ❓ 疑问6：「为什么Java这么啰嗦？」——给前端的一句话总结

```java
// 一个简单的"打印Hello"
public class HelloWorld {              // 1. 必须有个类
    public static void main(String[] args) { // 2. 必须有个main方法
        System.out.println("Hello");   // 3. 调用方式也是"类.方法"
    }
}
```

### 对比前端
```javascript
console.log("Hello");  // 一行搞定
```

### 原因——把话说清楚

> **Java 的"啰嗦" = 编译器的"安全"**。每多写一个字，编译器就能多检查一个潜在错误。

| Java啰嗦的地方 | 编译器帮你检查了什么 |
|---------------|-------------------|
| `public class Hello` | 文件名必须和类名一致，防止混乱 |
| `public static void main(String[] args)` | 方法签名必须精确匹配JVM规范 |
| `private String name;` | 其他类不能直接访问，封装安全 |
| `throws IOException` | 调用者必须处理可能的异常 |
| `List<String>` | 只能放String，放Integer编译报错 |
