## 🔷 6.1 Record——不可变DTO的终极形态

### 🧩 前端类比

> **Record ≈ TS的 `type User = { readonly id: number; readonly name: string }` + 自动生成 `equals`、`hashCode`、`toString`、构造器、getter——而你在前端要自己装一堆库来做这些事。**

前端你要做不可变DTO：
```typescript
// 前端——你需要借助各种工具
type User = {
  readonly id: number;
  readonly name: string;
  readonly email: string;
};
// 但 equals、toString、hashCode 全都要你自己写或用 lodash
```

### ☕ Java事实概念

**Record 是 Java 14 预览、Java 16 正式发布的"数据载体"类。** 你只需要声明字段，Java自动帮你生成：
- 全参**构造器**
- 字段 **getter**（注意：是 `name()` 不是 `getName()`！）
- `equals()` 和 `hashCode()`（基于所有字段值）
- `toString()`
- 所有字段自动 `private final`

### 📝 代码对比

| 前端 TS | Java 传统POJO（30行） | Java Record（1行） |
|---------|---------------------|-------------------|
| ```typescript<br>type User = {<br>  readonly id: number;<br>  readonly name: string;<br>};<br>// 无自动equals/toString<br>``` | ```java<br>public class UserDto {<br>  private final Long id;<br>  private final String name;<br>  public UserDto(Long id, String name) {<br>    this.id=id; this.name=name;<br>  }<br>  public Long getId() { return id; }<br>  public String getName() { return name; }<br>  @Override public boolean equals(...){...}<br>  @Override public int hashCode(){...}<br>  @Override public String toString(){...}<br>}<br>``` | ```java<br>public record UserDto(<br>  Long id,<br>  String name<br>) {}<br>``` |

### 💡 使用示例

```java
// 创建——和普通类一样new
var user = new UserDto(1L, "张三", "zhang@example.com");

// 读取——注意！不使用get前缀！
System.out.println(user.name());   // ✅ getter叫 name() 不是 getName()
System.out.println(user.id());     // ✅ id()

// toString——自动生成
System.out.println(user);  // UserDto[id=1, name=张三, email=zhang@example.com]

// equals——基于字段值比较
var user2 = new UserDto(1L, "张三", "zhang@example.com");
System.out.println(user.equals(user2));  // true ✅ 
```

### 🔴 核心差异（跟前端比）

| 方面 | 前端（TS） | Java Record |
|------|-----------|-------------|
| 创建对象 | `const u = { id:1, name:"张三" }` | `new UserDto(1L, "张三")` |
| 不可变性 | `readonly` 只防TS编译时，JS运行时随便改 | `final` 编译+运行时都不可改 |
| getter命名 | `u.name`（属性访问） | `user.name()`（方法调用！） |
| equals | 无内置，用 `_.isEqual()` | 自动基于所有字段 |
| 修改 | 用 spread: `{...u, name:"李四"}` | **没有setter**，只能new新的 |

## 🔷 6.2 Pattern Matching + Sealed Class——让 `switch` 脱胎换骨

### 🧩 前端类比

> **Pattern Matching ≈ TS 的 discriminated union（可辨识联合） + switch，但更强大——它自动提取字段，你不需要手动写类型守卫。**

### TS对比先行

```typescript
// TS——必须手动写 type guard
type Shape = 
  | { kind: 'circle'; radius: number }
  | { kind: 'rectangle'; width: number; height: number }
  | { kind: 'triangle'; base: number; height: number };

function calculateArea(shape: Shape): number {
  switch (shape.kind) {          // 手动判type
    case 'circle': 
      return Math.PI * shape.radius ** 2;  // 还得显式访问
    case 'rectangle': 
      return shape.width * shape.height;
    // 忘掉 triangle 了？编译器不报错！
  }
}
```

### ☕ Java事实概念

**Pattern Matching（模式匹配）** + **Sealed Class（密封类）** 是Java 17/21引入的完美组合：

- **`sealed`** 密封类/接口：明确告诉编译器"我的子类只有这几种"
- **`switch` + pattern matching**：直接解构出字段值，**不需要写 `instanceof` 和强制转换**
- **编译器穷举检查**：因为 `sealed` 限制了子类数量，switch 没有 `default` 也不会漏！

### 📝 代码对比

```java
// 1️⃣ 定义密封接口——Java 17
sealed interface Shape 
    permits Circle, Rectangle, Triangle {}  // 只允许这3个实现

// 2️⃣ 用 Record 实现——21行变3行
record Circle(double radius)       implements Shape {}
record Rectangle(double width, double height) implements Shape {}
record Triangle(double base, double height)    implements Shape {}

// 3️⃣ Pattern Matching——switch直接解构
public double calculateArea(Shape shape) {
    return switch (shape) {
        case Circle(double r)        -> Math.PI * r * r;          // r自动解构出来
        case Rectangle(double w, double h) -> w * h;              // w, h自动解构
        case Triangle(double b, double h)  -> b * h / 2;
        // ⭐ 不需要 default！编译器知道只有3种，不全就报错
    };
}
```

### 💥 对比震撼

```typescript
// TS版——手写到崩溃
function area(s: Shape) {
  if ('radius' in s) return Math.PI * (s as any).radius ** 2;
  // 手动cast，手动访问，容易漏case
}
```

```java
// Java版——编译器帮你穷举
return switch (shape) {
    case Circle(double r) -> Math.PI * r * r;
    case Rectangle(double w, double h) -> w * h;
    // 如果只写了上面两个，不写Triangle → ❌ 编译报错！
};
```

### 🟡 三大易错点

```java
// ❌ 错误1：sealed class 和子类必须在同一个模块/包内
// sealed interface Shape permits ExternalClass {}  // ❌ 编译报错

// ❌ 错误2：sealed 的子类必须用 sealed / non-sealed / final 标记
record Circle(double r) implements Shape {}     // ✅ 隐式 final
// 等价于：final class Circle ... 

// ❌ 错误3：别忘了 sealed 子类也可以是 sealed
sealed interface Shape permits Polygon, Circle {}
sealed interface Polygon extends Shape permits Rectangle, Triangle {}  // ✅
```

## 🔷 6.3 Text Blocks——Java版模板字符串

### 🧩 前端类比

> **Text Blocks ≈ JS 的模板字面量（反引号 ` `` `），但Java用的是三个双引号 `"""`。**

### 📝 代码对比

```typescript
// 前端——用反引号，爽
const json = `
{
  "name": "张三",
  "age": 25
}`;
```

```java
// Java传统写法——生不如死
String json = "{\n" +
              "  \"name\": \"张三\",\n" +
              "  \"age\": 25\n" +
              "}";

// Text Blocks——Java 15正式版
String json = """
    {
      "name": "张三",
      "age": 25
    }
    """;
```

### 💡 高级用法

```java
// 1️⃣ 配合 formatted()——像JS的 `${变量}`
String query = """
    SELECT id, name, email
    FROM users
    WHERE age > %d
    ORDER BY name
    """.formatted(18);
// 输出：WHERE age > 18

// 2️⃣ 写HTML
String html = """
    <html>
      <body>
        <h1>%s</h1>
        <p>%s</p>
      </body>
    </html>
    """.formatted("标题", "内容");

// 3️⃣ 行尾反斜杠——去掉换行（类似JS的续行）
String s = """
    hello \
    world""";
// 实际上是 "hello world"（连在一起）
```

### ⚠️ 踩坑提醒

```java
// ❌ 错误：开始 """ 后面不能跟字符
String s = """text""";    // ❌ 编译报错！"""后面必须换行

// ✅ 正确：
String s = """
    text""";   // ✅ 第一行结束符是 三个引号的位置

// ❌ 注意缩进：Text Block会把左边的公共缩进去掉
String s = """
    第一行
      第二行（缩进更深）
    第三行
    """;
// 输出：
// 第一行
//   第二行（缩进更深）
// 第三行
// 规则：取所有行中最小公共缩进来统一去掉
```

## 🔷 6.4 Virtual Threads——Java的"异步平权"

> ⏰ **这是下午2.5h的核心内容，也是Java 21最革命性的特性！**

### 🧩 前端类比

> **Virtual Thread ≈ Node.js 的 async/await 协程**，但它是操作系统级别的轻量线程——**你写的是同步代码，但底层是异步执行的。**

### 前端噩梦 vs Java的解药

```typescript
// 前端——async/await 会传染整个调用链
async function handleRequest(userId: string) {
  const user = await db.findUser(userId);   // await 传染
  const orders = await db.findOrders(user.id); 
  await sendEmail(user.email, orders);
  // 调了这个函数的人，也得 await 
  // 整个调用链全变成 async... 太痛苦了
}
```

```java
// Virtual Threads——不需要 async/await！
void handleRequest(long userId) {
    var user = userRepo.findById(userId).get();     // 看着像阻塞，实际不阻塞
    var orders = orderRepo.findByUserId(user.id);   // 会"挂起"但释放底层线程
    emailService.send(user.email(), orders);         // 底层自动处理
    // 调用方不需要任何 async/await，代码是同步的！
}
```

### ☕ Java事实概念

**Virtual Thread（虚拟线程）是 Java 19 预览、Java 21 正式发布的轻量级线程。** 

核心原理：
- 传统**平台线程**（Platform Thread）≈ 操作系统线程，每个约 1MB，最多几千个
- **虚拟线程**（Virtual Thread）≈ 在Java层面模拟的线程，每个约几百字节，可以开**百万级**
- 当虚拟线程执行到阻塞操作（如读数据库、调API）时，JVM自动"挂起"它，把底层平台线程让给其他虚拟线程
- **你写的代码看起来是同步阻塞的，但底层是异步非阻塞的！**

### 📝 代码对比：传统线程 vs Virtual Thread

```java
// 🔴 传统方式：开1000个线程？内存直接爆炸
for (int i = 0; i < 1000; i++) {
    new Thread(() -> {
        // 每个平台线程消耗 ~1MB 栈内存
        doSomething();
    }).start();
}
// 开1万个？你的电脑可能直接挂了
```

```java
// 🟢 Virtual Threads：开100万个？没问题！
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 1_000_000; i++) {
        executor.submit(() -> {
            // 每个虚拟线程消耗 ~几百字节！
            doSomething();
        });
    }
}  // 自动等待所有完成
```

```java
// 更简单的写法
Thread.startVirtualThread(() -> {
    var data = fetchFromDB();        // "阻塞"调用，但底层会挂起
    var result = processData(data);
    saveToDB(result);
});
```

### 🆚 三向对比表

| 特性 | 传统平台线程 | Virtual Thread | Node.js 异步 |
|------|-------------|----------------|-------------|
| 创建成本 | ~1MB/线程 | **~几百字节**/线程 | 极低（回调） |
| 最大数量 | 几千个 | **百万级** | 受限于事件循环 |
| 编程模型 | 同步阻塞 | **同步阻塞**（底层异步） | `async/await` 传染 |
| 调试体验 | 还行 | **简单**（同步代码栈） | 地狱级异步调用栈 |
| try-catch | 正常 | 正常 | 异步异常难捕获 |
| 适合场景 | 少量长任务 | 大量短任务（IO密集型） | 大量短任务 |

### 💡 Virtual Thread 对前端的革命性意义

> **对前端开发者来说**：Virtual Threads 是 Java 版的"你写同步代码，享受异步性能"。不像 Node.js 需要 `async/await` 传染整个调用链。

```typescript
// Node.js——一个 async 传染所有
async function handler(req, res) {
  const user = await db.getUser(req.params.id);  // async 
  await emailService.send(user);                  // async 无处不在
  res.json({ ok: true });
}
// 能传染到 main()... 整个项目全得 async
```

```java
// Java Virtual Thread——不需要async
void handler(long userId) {
    var user = userRepo.findById(userId);  // 看着像同步
    emailService.send(user);               // 还是同步写法
    // 底层自动异步，但你不需感知
}
```

public class User {
    private Long id;
    private String username;
    private String email;

## 📝 练习2：用 sealed class + pattern matching 实现表达式求值器（40min）

### 作业要求
实现一个简单的表达式求值器，能处理整数、加法、乘法运算。

### 完整代码

```java
// 📁 Expr.java —— 定义密封接口
package com.demo;

// 密封接口：表达式只能是 IntegerExpr、AddExpr、MultiplyExpr 三种
sealed interface Expr 
    permits IntegerExpr, AddExpr, MultiplyExpr {}

// Record实现
record IntegerExpr(int value)       implements Expr {}
record AddExpr(Expr left, Expr right)      implements Expr {}
record MultiplyExpr(Expr left, Expr right) implements Expr {}
```

```java
// 📁 EvalDemo.java —— 模式匹配求值器
package com.demo;

public class EvalDemo {
    
    public static int eval(Expr expr) {
        return switch (expr) {
            case IntegerExpr(int v)                -> v;
            case AddExpr(Expr l, Expr r)           -> eval(l) + eval(r);
            case MultiplyExpr(Expr l, Expr r)      -> eval(l) * eval(r);
            // sealed 保证了穷举，不需要 default！
        };
    }
    
    public static void main(String[] args) {
        // 构造表达式: (1 + 2) * 3 = 9
        // 抽象语法树：
        //     MultiplyExpr
        //      /        \
        //   AddExpr      3
        //   /     \
        //  1       2
        Expr expr = new MultiplyExpr(
            new AddExpr(new IntegerExpr(1), new IntegerExpr(2)),
            new IntegerExpr(3)
        );
        
        System.out.println("(1 + 2) * 3 = " + eval(expr));  // 输出: 9
        
        // 更多测试
        System.out.println("1 = " + eval(new IntegerExpr(1)));     // 1
        System.out.println("1 + 2 = " + eval(new AddExpr(
            new IntegerExpr(1), new IntegerExpr(2)
        )));  // 3
    }
}
```

### 💡 扩展思考

```java
// 如果要支持减法、除法怎么办？
sealed interface Expr permits IntegerExpr, AddExpr, MultiplyExpr, 
    SubtractExpr, DivideExpr {}  // 加两个子类型

record SubtractExpr(Expr left, Expr right) implements Expr {}
record DivideExpr(Expr left, Expr right) implements Expr {}

// 编译器会强制你必须加上新分支：
public static int eval(Expr expr) {
    return switch (expr) {
        case IntegerExpr(int v)                -> v;
        case AddExpr(Expr l, Expr r)           -> eval(l) + eval(r);
        case MultiplyExpr(Expr l, Expr r)      -> eval(l) * eval(r);
        // ❌ 编译报错！缺少 SubtractExpr 和 DivideExpr 分支！
    };
}
// 加上了：
public static int eval(Expr expr) {
    return switch (expr) {
        case IntegerExpr(int v)                -> v;
        case AddExpr(Expr l, Expr r)           -> eval(l) + eval(r);
        case MultiplyExpr(Expr l, Expr r)      -> eval(l) * eval(r);
        case SubtractExpr(Expr l, Expr r)      -> eval(l) - eval(r);
        case DivideExpr(Expr l, Expr r)        -> eval(l) / eval(r);
    };
}
// 🎯 编译器帮你穷举检查——再也不会忘记加分支了！
```

## 📝 练习4：用 Virtual Threads 创建10000个并发任务，对比传统线程（30min）

### 完整代码

```java
// 📁 VirtualThreadDemo.java
package com.demo;

import java.util.concurrent.*;
import java.time.*;

public class VirtualThreadDemo {
    
    // 模拟一个IO操作（如查数据库、调API）
    static void ioTask(int id) {
        try {
            // 模拟阻塞（比如网络IO等待100ms）
            Thread.sleep(100);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
    
    public static void main(String[] args) throws Exception {
        final int TASK_COUNT = 10000;  // 任务数量
        
        // ========== 1. 传统线程测试 ==========
        System.out.println("=== 🚫 传统平台线程（只开100个，开1万个内存会爆） ===");
        long start = System.currentTimeMillis();
        
        var platformThreads = new Thread[100];  // 只敢开100个
        for (int i = 0; i < 100; i++) {
            final int id = i;
            platformThreads[i] = new Thread(() -> ioTask(id));
            platformThreads[i].start();
        }
        for (Thread t : platformThreads) {
            t.join();  // 等待所有线程完成
        }
        long platformTime = System.currentTimeMillis() - start;
        System.out.println("平台线程 (100个) 耗时: " + platformTime + "ms");
        
        // ========== 2. Virtual Thread 测试 ==========
        System.out.println("\n=== ✅ Virtual Thread（开1万个，轻松） ===");
        start = System.currentTimeMillis();
        
        try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
            for (int i = 0; i < TASK_COUNT; i++) {
                final int id = i;
                executor.submit(() -> ioTask(id));
            }
        }  // try-with-resources 自动等待所有完成
        
        long vtTime = System.currentTimeMillis() - start;
        System.out.println("Virtual Thread (" + TASK_COUNT + "个) 耗时: " + vtTime + "ms");
        
        // ========== 3. 性能对比 ==========
        System.out.println("\n=== 📊 对比结论 ===");
        System.out.println("传统线程开1万个 → 内存可能溢出或卡死");
        System.out.println("Virtual Thread开1万个 → 轻松完成，耗时约" + vtTime + "ms");
        System.out.println("🎯 Virtual Thread 创建了 " + TASK_COUNT 
            + " 个并发任务，每个sleep 100ms，总耗时约 " + vtTime + "ms");
        System.out.println("（并行执行的！不是串行！总耗时≈100ms + 少量调度开销）");
    }
}
```

### 💡 如果你电脑配置高，可以挑战更大数量

```java
// 尝试 100 万个虚拟线程！
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 1_000_000; i++) {
        executor.submit(() -> {
            // 每个只做一点点事
            int x = 0;
            for (int j = 0; j < 100; j++) x += j;
        });
    }
}
// 如果是传统线程...你的电脑可能会冒烟 💨
```

## 🔴 难点1：Record的 getter 为什么是 `name()` 而不是 `getName()`？

### 你的疑问
> "我习惯了前端的 `user.name` 属性和 Java 传统的 `user.getName()`，Record 突然来个 `user.name()` 太奇怪了！"

### 深度解释

**Record的设计哲学是"数据载体"而不是"JavaBean"。**

传统 Java 的 `getXxx()` 命名约定源于 JavaBean 规范（1996年），当时是为了让 IDE 的工具能自动识别属性。但 Record 的目标是**精简**：

```java
// 传统Java Bean
public class User {
    private String name;
    public String getName() { return name; }  // get前缀
}

// Record——直截了当
public record User(String name) {}
// 自动生成：public String name() { return this.name; }
```

**记忆窍门：** Record 的 getter 名 = 字段名 + `()`。`user.name()` 就是读取 `name` 字段。没有 `get`，没有 `set`，字段说啥就是啥。

### 为什么这样设计？
- 前端：`user.name`（属性访问）
- Java传统：`user.getName()`（方法调用）
- Java Record：`user.name()`（方法调用，但名字直接 = 字段名）
- **Record 就是告诉你："我不玩花活，叫啥就是啥"**

---

## 🔴 难点2：Record 没有 setter，那我要修改怎么办？

### 你的疑问
> "前端我用 `user.name = '新名字'` 或者 `{...user, name: '新名字'}` 随便改。Record 不能修改，我怎么更新数据？"

### 深度解释

**Record 的设计理念就是"不可变"（immutable）。** 如果要修改，就创建一个新的 Record：

```java
// Record——没有setter，不能修改
var user = new User(1L, "张三", "zhang@example.com", 25);
// user.setName("李四");  // ❌ 不存在！

// ✅ 正确的做法：创建一个新的
var updatedUser = new User(
    user.id(),           // id 不变
    "李四",              // 新名字
    user.email(),       // email 不变
    user.age()          // age 不变
);
```

### 前端类比

```typescript
// 前端不可变更新
const user = { id: 1, name: '张三', email: '...', age: 25 };
const updated = { ...user, name: '李四' };  // 创建新对象

// Java Record——同样的概念！创建新Record
var updated = new User(user.id(), "李四", user.email(), user.age());
```

### 如果字段太多怎么办？

```java
// Java 16+ 的 with 操作符（预览功能，可能已正式）
// var updated = user with { name = "李四" };  // 🚧 实验性

// 或者用 Builder 模式配合 Record：
public record User(Long id, String username, String email, Integer age) {
    public User withUsername(String newUsername) {
        return new User(this.id, newUsername, this.email, this.age);
    }
}
// 使用：
var updated = user.withUsername("李四");
```

### 为什么 Record 要设计成不可变？
1. **线程安全**：不可变对象天然线程安全，不需要加锁
2. **可缓存**：hashCode 可以缓存，提高 Map 性能
3. **无副作用**：传参时不用担心被意外修改
4. **适合做DTO**：数据传输对象通常只需要读，不需要改

## 🔴 难点3：`sealed` 的关键字和限制

### 你的疑问
> "`sealed interface Shape permits Circle, Rectangle, Triangle {}` 这行代码看懵了。为什么要有 sealed？"

### 深度解释

**`sealed`（密封） ≈ 前端的 `export type Shape = Circle | Rectangle | Triangle`**

前端的 discriminated union 天然就是"密封"的——可辨识联合的成员是固定的。但 Java 的 class/interface 默认是"开放"的——任何人都可以继承。

```java
// 没有 sealed——任何类都能继承 Shape
interface Shape {}
class Circle implements Shape {}  // ✅ 谁都可以
class Rectangle implements Shape {}  // ✅ 随便加
class MySecretShape implements Shape {}  // ✅ 别人也能偷偷加
```

```java
// 有了 sealed——只有允许的才能继承
sealed interface Shape permits Circle, Rectangle, Triangle {}
// 其他类不能再 implements Shape 了！

record Circle(double radius) implements Shape {}     // ✅
record Rectangle(double w, double h) implements Shape {}  // ✅
record Pentagon(double side) implements Shape {}    // ❌ 编译报错！不在permits列表里
```

### sealed 的三个要求

```java
// 要求1：permits 不能为空
sealed interface A {}  // ❌ 没有permits但也没有子类？编译报错

// 要求2：子类必须用 sealed / non-sealed / final 标记
sealed interface Shape permits Circle {}
// Circle 必须是 sealed / non-sealed / final 之一：
final record Circle(double r) implements Shape {}   // ✅ final
// 或者 non-sealed（允许任何人继承Circle）：
non-sealed class Circle implements Shape {}          // ✅ 开放了

// 要求3：子类和密封类必须在同一个模块或包内（防止别人乱继承）
```

## 🔴 难点4：Pattern Matching 的 `case Circle(double r)` 是什么语法？

### 你的疑问
> "`case Circle(double r) -> ...` 这看起来好像在构造一个 Circle，怎么又是解构了？"

### 深度解释

**这是"解构模式"（Destructuring Pattern）。** 它和构造相反——构造是把值装进去，解构是把值拆出来。

```java
// 构造（把值装进去）——你Day 2学的
var circle = new Circle(5.0);   // 把 5.0 "装进" Circle 里面

// 解构（把值取出来）——Pattern Matching
case Circle(double r) -> ...    // 把 Circle 里的 radius "取出来" 赋值给 r
```

### 前端类比

```typescript
// TS的解构赋值——你天天用
const { radius } = circle;  // 把 circle.radius 取出来

// Java Pattern Matching 的解构
case Circle(double r) -> ...  // 把 circle.radius() 取出来，赋值给 r
```

### 更深层的理解

```java
// ❌ 没有 Pattern Matching 的写法
if (shape instanceof Circle) {
    Circle c = (Circle) shape;          // 手动强制转换
    double r = c.radius();              // 手动调用getter
    return Math.PI * r * r;
}

// ✅ Pattern Matching（Java 16简化版）
if (shape instanceof Circle c) {       // 自动转换
    return Math.PI * c.radius() * c.radius();
}
// 但 if 里还要手动访问 radius()

// ✅ Pattern Matching + switch（Java 21完全体）
return switch (shape) {
    case Circle(double r) -> Math.PI * r * r;  // 直接解构！
};
```

### 嵌套解构

```java
// 如果括号表达式是 Add(1, Add(2, 3)) 呢？
// Pattern Matching 可以嵌套解构！

public static int eval(Expr expr) {
    return switch (expr) {
        case IntegerExpr(int v) -> v;
        case AddExpr(IntegerExpr(int a), IntegerExpr(int b)) -> a + b;
        // 只匹配 "两个整数相加" 的模式
        
        case AddExpr(Expr l, Expr r) -> eval(l) + eval(r);
        // 通用匹配 "任意表达式相加"
    };
}
```

## 🔴 难点5：Virtual Thread 到底是怎么回事？和 Node.js 的 async/await 区别在哪里？

### 你的疑问
> "Node.js 说事件循环是单线程的，Java Virtual Thread 说可以开百万个。这两者到底什么关系？谁更厉害？"

### 深度解释

这是 **Day 6 最重要也最容易混淆的概念！** 我用一个餐馆的比喻讲清楚：

### 🍜 餐馆比喻

**场景：** 你有10张餐桌（类比CPU核心数），每张桌子只能坐一个人（线程）。

**传统 Java（平台线程）：**
```
你雇了10个厨师，每人站一张桌子后面。
来一个客人 → 分配给一个厨师 → 厨师从点餐、做菜到上菜全程陪同
客人等菜时发呆，厨师也闲着但不能走开
→ 要服务1000个客人？雇1000个厨师！成本太高了！
```

**Node.js（事件循环）：**
```
你只雇了1个厨师，在厨房里。
来了100个客人 → 厨师一个个问"你要什么？"
点完后去炒菜 → 炒完一个端出来给客人 → 再去问下一个客人
→ 厨师很忙，但炒菜（CPU）和等客人思考（IO）交替进行
→ 问题：厨师最多同时炒一个菜，炒菜慢了后面全得等
```

**Java Virtual Thread：**
```
你雇了10个厨师（平台线程），但每个厨师可以分身成100个虚拟厨师。
来了10000个客人 → 前台登记（虚拟线程）
客人点餐时（IO等待）→ 虚拟厨师自动让出位置 → 去服务另一个客人
等菜做好了 → 虚拟厨师回来继续服务
→ 看起来像有10000个厨师在服务，但实际只有10个人在做事
→ 客人的体验：每个人都有一个专属厨师！（写代码的体验）
```

### 代码对比

```typescript
// Node.js——每个请求一个 async 函数
app.get('/user/:id', async (req, res) => {  // async 传染
  const user = await db.getUser(req.params.id);  // await 到处是
  const orders = await db.getOrders(user.id);
  res.json({ user, orders });
});
// 看起来不像是"每个用户一个线程"，而是"一个线程切来切去"
```

```java
// Java Virtual Thread——每个请求看起来是一个独立线程
app.get("/user/{id}", ctx -> {
    // 不用 async，不用 await
    var user = userRepo.findById(ctx.pathParam("id"));  // 阻塞？
    var orders = orderRepo.findByUserId(user.id());     // 阻塞？
    ctx.json(Map.of("user", user, "orders", orders));
    // 代码看起来"阻塞"，底层JVM自动挂起/恢复
});
// 看起来像"每个请求有一个专属线程"！
```

### 🆚 本质区别总结

| 维度 | Node.js 异步 | Java Virtual Thread |
|------|-------------|-------------------|
| 编程模型 | **显式异步**（必须写 `async/await`） | **透明异步**（代码像同步） |
| 函数传染 | ✅ **会传染**——调async函数必须也async | ❌ **不传染**——调VT方法不需要任何特殊处理 |
| 调试 | 🔴 地狱——异步调用栈支离破碎 | 🟢 简单——就是普通堆栈 |
| 底层实现 | 单线程事件循环 + 回调队列 | 虚拟线程挂载在平台线程上 |
| 本质 | **协作式多任务**（你得主动yield） | **抢占式多任务**（JVM自动调度） |

### 🎯 给前端开发者的关键认知

> **Virtual Thread 不是"Java版的async/await"——它比那更高级。它让你**不需要**写异步代码就享受异步性能。** 

想象一下，如果 Node.js 不需要 `async/await`，每个请求都像同步代码一样写，但底层自动异步——那就是 Virtual Thread。

## 🔴 难点6：Text Blocks 的缩进规则

### 你的疑问
> "Text Block 输出的字符串到底缩进多少？最左边的空格会不会保留？"

### 深度解释

**Java 的 Text Block 会去掉"公共缩进"（common leading white space）。**

```java
String s = """
    第一行
      第二行（缩进更多）
    第三行
    """;
// 实际的字符串内容（公共缩进被去掉）：
// "第一行\n" +
// "  第二行（缩进更多）\n" +
// "第三行\n"
```

### 规则详解

```java
// 规则1：结束 """ 的位置决定基准线
String s = """
    你好
    世界
    """;  
// 结束 """在缩进4格的位置 → 公共缩进是4个空格
// 输出：
// 你好
// 世界

// 规则2：某行缩进少于公共缩进 → 以该行最小缩进为准
String s = """
    第一行
  第二行（缩进更少）
    第三行
    """;
// 公共缩进 = 2（第二行的缩进）
// 输出：
//   第一行（2格缩进去掉后，还剩2格）
// 第二行（缩进更少）
//   第三行

// 规则3：空白行不算
String s = """
    第一行

    第三行
    """;
// 空白行不影响公共缩进计算
```

## 🔴 难点7：Virtual Thread 的 `try-with-resources` 是什么？

### 你的疑问
> "`try (var executor = Executors.newVirtualThreadPerTaskExecutor()) { ... }` 这个语法怎么看着不像 try-catch？"

### 深度解释

**`try-with-resources` ≈ 前端的 `using` / 自动调用 `dispose()`**

```typescript
// TypeScript（如果支持 using）
// using resource = getResource();
// 自动调用 resource.dispose()
```

```java
// 传统写法——手动关
ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor();
try {
    // 提交任务...
} finally {
    executor.shutdown();  // 手动关，容易忘！
}
```

```java
// try-with-resources——自动关
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    // 提交任务...
    for (int i = 0; i < 10000; i++) {
        executor.submit(() -> doSomething());
    }
}
// 离开 try 块时，自动调用 executor.close()，它会等待所有任务完成再关闭
```

**这就是 `try-with-resources` 的作用：**
- 任何实现了 `AutoCloseable` 接口的类，都可以用在 `try(...)` 里
- 离开 try 块时**自动调用 close()**
- ≈ 前端的 `using` 语法（C#/TypeScript 的实验性特性）
