# 📝 练习1：模拟读取文件——体会检查型异常的声明和处理

> **难度：⭐⭐ | 核心概念：Checked Exception（检查型异常）**

## 🧩 前端类比

**JS的"爱怎么样怎么样" vs Java的"必须白纸黑字写清楚"**

```typescript
// 前端：想抛就抛，想接就接
function parseJSON(str: string) {
    return JSON.parse(str);  // 可能抛，也可能不抛，没人管你
}

// 调用方：
parseJSON("invalid json");  // 不catch也行，运行时报错
```

```java
// Java：你必须"签字画押"
public static JSONObject parseJSON(String str) throws JSONException {
    return new JSONObject(str);  // throws 声明，承认"我会出问题"
}

// 调用方：必须 catch
try {
    parseJSON("invalid json");
} catch (JSONException e) {
    // 你不写这个？编译报错！
}
```

## 💻 完整代码

```java
// 文件名: FileReadDemo.java
import java.io.*;

public class FileReadDemo {
    
    public static void main(String[] args) {
        System.out.println("===== 练习1：模拟读取文件 =====");
        
        // 方式1：在main里try-catch处理
        try {
            String content = readFile("test.txt");
            System.out.println("文件内容: " + content);
        } catch (IOException e) {
            System.out.println("❌ 读取文件失败: " + e.getMessage());
            // e.printStackTrace();  // 打印完整堆栈（类似前端的 console.error(e)）
        }
        
        System.out.println("程序继续执行... 不会崩！✅");
        
        // 方式2：不处理，直接往上抛（给JVM）
        // 你看下面这行——如果去掉注释，main方法签名必须加 throws IOException
        // String content2 = readFile("nonexist.txt");
    }
    
    /**
     * 模拟读取文件
     * 👆 注意方法签名上的 throws IOException！
     * 这就是"检查型异常"——调用方必须处理，否则编译不通过
     */
    public static String readFile(String path) throws IOException {
        // 模拟：文件不存在时抛异常
        File file = new File(path);
        if (!file.exists()) {
            // throw 关键字和JS的 throw 用法完全一样
            throw new IOException("文件不存在: " + path);
        }
        
        // 用 BufferedReader 读取文件（标准写法）
        BufferedReader reader = new BufferedReader(new FileReader(file));
        String line = reader.readLine();
        reader.close();
        return line;
    }
}
```

## ❓ 练习1 的疑问点 & 难点

### 🔴🔴🔴 难点1（最大文化冲击）：`throws IOException` 到底是个啥？

这是**Java独有的、JS完全没有**的东西。

```java
// 写法
public static String readFile(String path) **throws IOException** {
```

> 🧩 **前端类比**：想象你写一个函数，在函数签名上就写清楚"我这个函数可能会崩"：
> ```typescript
> function readFile(path: string): string **throws IOException** {
>   // 调用方看到你这签名，就知道必须做错误处理
> }
> ```

**本质**：`throws IOException` 就是方法在"签字画押"——我承认我可能会抛出这种异常，调用方你看着办。

### 🔴 难点2：`IOException` 为什么强制我们处理？（而 `RuntimeException` 不用）

看这张图就懂了：

| 异常类型 | Java叫法 | 前端类比 | 是否强制处理 |
|---------|---------|---------|------------|
| `IOException` | 检查型异常 | 编译时"你代码有问题" | **必须** try-catch 或 throws |
| `NullPointerException` | 非检查型异常（运行时） | 运行时"哦豁崩了" | **不用**，但跑起来可能会崩 |

```java
// 非检查型异常 —— 不用 throws 声明，不用 try-catch
public static void riskyMethod() {
    // 这行编译能过，运行时会崩
    String s = null;
    s.length();  // NullPointerException 运行时才报
}

// 检查型异常 —— 必须声明 throws
public static void safeMethod() throws IOException {
    // 这行如果不处理，编译直接报错
    throw new IOException("出错了");  // 编译报错！必须声明 throws
}
```

### 🟡 避坑：`try-catch-finally` 和JS的区别

```java
// ✅ Java的 try-catch-finally 和JS 99%一样
try {
    // 可能出问题的代码
} catch (IOException e) {
    // 处理异常
} finally {
    // 不管有没有异常，都会执行
    // ≈ JS的 finally
}
```

**唯一区别**：Java的 `catch` 可以**按顺序捕获多个异常**：

```java
try {
    // ...
} catch (IOException e) {
    // 文件相关异常
} catch (SQLException e) {
    // 数据库相关异常
}
// 就像前端的多个 catch：
// .catch(err => { if (err instanceof IOError) ... })
```

### 🧪 必做实验：main方法签名上加 throws

修改 `main` 方法，把代码改成**不catch，往上抛**：

```java
public static void main(String[] args) **throws IOException** {  // 👈 加 throws
    String content = readFile("test.txt");  // 现在不用try-catch了
    System.out.println(content);
}
```

编译运行：
```bash
javac FileReadDemo.java && java FileReadDemo
```

你会看到什么？**程序直接崩了！JVM打印了完整的错误堆栈然后退出**。这就是"把异常抛给了JVM"——相当于前端没catch住错误，浏览器控制台报红。

# 📝 练习2：自定义异常类

> **难度：⭐ | 核心概念：继承 RuntimeException**

## 🧩 前端类比

```typescript
// 前端自定义错误
class ValidationError extends Error {
    constructor(message: string, public field: string) {
        super(message);
        this.name = 'ValidationError';
    }
}

throw new ValidationError('用户名不能为空', 'username');
```

```java
// Java自定义异常——几乎一样！
public class ValidationException extends RuntimeException {
    private String field;
    
    public ValidationException(String message, String field) {
        super(message);
        this.field = field;
    }
    
    public String getField() {
        return field;
    }
}
```

## 💻 完整代码

```java
// 文件名: BusinessException.java
/**
 * 自定义业务异常
 * 继承 RuntimeException → 非检查型异常，不强制调用方处理
 * 
 * 🧩 前端类比：class BusinessError extends Error
 */
public class BusinessException extends RuntimeException {
    
    // 错误码（类似前端的 error.code）
    private String errorCode;
    
    // 构造器1：只传消息
    public BusinessException(String message) {
        super(message);
        this.errorCode = "UNKNOWN";
    }
    
    // 构造器2：传消息 + 错误码
    public BusinessException(String message, String errorCode) {
        super(message);
        this.errorCode = errorCode;
    }
    
    // 构造器3：传消息 + 原因（异常链）
    public BusinessException(String message, Throwable cause) {
        super(message, cause);
        this.errorCode = "UNKNOWN";
    }
    
    public String getErrorCode() {
        return errorCode;
    }
}
```

```java
// 文件名: BusinessExceptionDemo.java
/**
 * 演示自定义异常的使用
 */
public class BusinessExceptionDemo {
    
    public static void main(String[] args) {
        System.out.println("===== 练习2：自定义异常 =====");
        
        // 场景：模拟用户注册时的校验
        try {
            registerUser("", "123456");
        } catch (BusinessException e) {
            System.out.println("❌ 注册失败！");
            System.out.println("   错误码: " + e.getErrorCode());
            System.out.println("   错误信息: " + e.getMessage());
        }
        
        System.out.println("程序继续执行 ✅");
    }
    
    /**
     * 模拟用户注册
     * 注意：这个方法没有 throws BusinessException
     * 因为 BusinessException 继承 RuntimeException（非检查型）
     */
    public static void registerUser(String username, String password) {
        if (username == null || username.isEmpty()) {
            throw new BusinessException("用户名不能为空", "USERNAME_EMPTY");
        }
        
        if (password == null || password.length() < 6) {
            throw new BusinessException("密码长度不能少于6位", "PASSWORD_TOO_SHORT");
        }
        
        System.out.println("✅ 用户注册成功: " + username);
    }
}
```

## ❓ 练习2 的疑问点 & 难点

### 🔴 难点：为什么继承 `RuntimeException` 而不是 `Exception`？

```java
// 方案A：继承 Exception → 检查型异常，调用方必须处理
public class MyException extends Exception { }  // ❌ 调用方被迫try-catch

// 方案B：继承 RuntimeException → 非检查型，调用方可选
public class MyException extends RuntimeException { }  // ✅ 调用方自由
```

> 🧩 **前端类比**：
> - `extends Exception` ≈ 你在TS函数上写了 `@throws` 注解，所有调用方必须try-catch
> - `extends RuntimeException` ≈ 普通 `throw new Error()`，调用方爱catch不catch

**实践中**：99%的自定义异常都继承 `RuntimeException`。因为如果继承 `Exception`，你的业务代码每一层都要 `throws`，调用链会变得极度冗长。

### 🟡 避坑：异常链（cause）

```java
try {
    // 底层IO异常
    readFile("config.txt");
} catch (IOException e) {
    // 包装成业务异常，保留原始原因
    throw new BusinessException("读取配置文件失败", e);  // 👈 e 就是 cause
}

// 使用方可以拿到原始异常
catch (BusinessException e) {
    Throwable cause = e.getCause();  // 👈 拿到原始的 IOException
    System.out.println("原始原因: " + cause.getMessage());
}
```

> 🧩 **前端类比**：就像 `.catch(err => { throw new WrapperError('错了', { cause: err }) })`

# 📝 练习3：用 Optional 重构 null 判断

> **难度：⭐⭐⭐ | 核心概念：Optional 链式调用**

## 🧩 前端类比

```typescript
// 前端——可选链 + 空值合并
const city = user?.address?.city ?? '未知城市';
```

```java
// Java——Optional链式调用
String city = Optional.ofNullable(user)
    .map(User::getAddress)
    .map(Address::getCity)
    .orElse("未知城市");
```

**思维转换**：前端用 `?.` 和 `??` 操作符"偷懒"，Java用 `Optional` 的 `map/flatMap/orElse` 方法链"显式处理空"。

## 💻 完整代码

```java
// 文件名: OptionalDemo.java
import java.util.*;

/**
 * 练习3：用 Optional 重构 null 判断
 * 
 * 场景：根据用户ID查找用户，然后获取其收货地址的城市名
 */
public class OptionalDemo {
    
    // —— 模拟数据库 ——
    private static Map<Long, User> userDB = new HashMap<>();
    
    static {
        // 初始化一些数据
        userDB.put(1L, new User(1L, "张三", new Address("北京市", "海淀区")));
        userDB.put(2L, new User(2L, "李四", null));  // 👈 李四没有地址
        // 3号用户不存在
    }
    
    public static void main(String[] args) {
        System.out.println("===== 练习3：Optional 重构 null 判断 =====");
        
        // —— 传统 null 判断方式（你以前在JS里怎么做的）——
        System.out.println("\n--- 传统方式（if-null判断）---");
        String city1 = getCityNameTraditional(1L);
        System.out.println("用户1的城市: " + city1);
        
        String city2 = getCityNameTraditional(2L);
        System.out.println("用户2的城市: " + city2);
        
        String city3 = getCityNameTraditional(3L);
        System.out.println("用户3的城市: " + city3);
        
        // —— Optional 链式调用方式 ——
        System.out.println("\n--- Optional方式（链式调用） ---");
        String city4 = getCityNameOptional(1L);
        System.out.println("用户1的城市: " + city4);
        
        String city5 = getCityNameOptional(2L);
        System.out.println("用户2的城市: " + city5);
        
        String city6 = getCityNameOptional(3L);
        System.out.println("用户3的城市: " + city6);
        
        // —— Optional 的其他常用操作 ——
        System.out.println("\n--- Optional 其他操作 ---");
        
        // orElseThrow：找不到就抛异常
        try {
            User user = findById(3L)
                .orElseThrow(() -> new RuntimeException("用户不存在！"));
        } catch (RuntimeException e) {
            System.out.println("❌ " + e.getMessage());
        }
        
        // ifPresent：存在时才执行
        findById(1L).ifPresent(user -> 
            System.out.println("✅ 找到用户: " + user.getName())
        );
        
        // orElseGet：延迟计算的默认值
        User defaultUser = findById(3L)
            .orElseGet(() -> new User(0L, "默认用户", null));
        System.out.println("默认用户: " + defaultUser.getName());
    }
    
    /**
     * 根据ID查找用户
     * ✅ 返回 Optional<User>
     * 
     * 🧩 前端类比：
     * function findById(id: number): User | null
     * 但Java强迫你返回 Optional<User>，告诉调用方"可能没有"
     */
    public static Optional<User> findById(Long id) {
        // Optional.ofNullable：如果userDB.get(id) 返回null，就包装成 Optional.empty()
        return Optional.ofNullable(userDB.get(id));
    }
    
    /**
     * ❌ 传统方式：到处都是 if-null 判断
     * 这是你前端习惯的写法，但Java里会非常冗长
     */
    public static String getCityNameTraditional(Long userId) {
        // 第一层if：用户是否存在
        User user = userDB.get(userId);  // 可能 null
        if (user == null) {
            return "未知城市";
        }
        
        // 第二层if：地址是否存在
        Address address = user.getAddress();  // 可能 null
        if (address == null) {
            return "未知城市";
        }
        
        // 第三层if：城市名是否存在
        String city = address.getCity();  // 理论上不为null，但以防万一
        if (city == null) {
            return "未知城市";
        }
        
        return city;
    }
    
    /**
     * ✅ Optional方式：链式调用，没有if-null
     * 
     * 🧩 前端类比：
     * const city = findById(id)
     *   ?.address
     *   ?.city ?? '未知城市'
     */
    public static String getCityNameOptional(Long userId) {
        return findById(userId)                // Optional<User>
            .map(User::getAddress)             // Optional<Address> —— 如果User为null，这里就是empty
            .map(Address::getCity)             // Optional<String>  —— 如果Address为null，这里就是empty
            .orElse("未知城市");                // String —— 如果上面任何一步为null，返回默认值
    }
}

/**
 * 用户类
 */
class User {
    private Long id;
    private String name;
    private Address address;
    
    public User(Long id, String name, Address address) {
        this.id = id;
        this.name = name;
        this.address = address;
    }
    
    public Long getId() { return id; }
    public String getName() { return name; }
    public Address getAddress() { return address; }
}

/**
 * 地址类
 */
class Address {
    private String city;
    private String district;
    
    public Address(String city, String district) {
        this.city = city;
        this.district = district;
    }
    
    public String getCity() { return city; }
    public String getDistrict() { return district; }
}
```

## ❓ 练习3 的疑问点 & 难点

### 🔴🔴🔴 难点1（最核心）：`Optional.ofNullable` vs `Optional.of` vs `Optional.empty()`

这是新手最懵逼的地方——三个创建方式啥区别？

```java
// 1. Optional.of(value) —— value 不能为 null！
Optional<User> opt1 = Optional.of(user);        // ✅ user不为null
Optional<User> opt2 = Optional.of(null);        // ❌ 运行时报 NullPointerException！

// 2. Optional.ofNullable(value) —— value 可以为 null
Optional<User> opt3 = Optional.ofNullable(user);  // ✅ user为null时变成empty
Optional<User> opt4 = Optional.ofNullable(null);  // ✅ 不会报错，返回empty

// 3. Optional.empty() —— 返回一个空的Optional
Optional<User> opt5 = Optional.empty();           // ≈ 前端的 null
```

> 🧩 **前端类比**：
> - `Optional.of(x)` ≈ 你确定 x 一定不为 null
> - `Optional.ofNullable(x)` ≈ `x ?? null`（不确定）
> - `Optional.empty()` ≈ `null`

**黄金法则**：**99%的情况下用 `Optional.ofNullable()`**。只有在你**100%确定**值不为null时才用 `Optional.of()`。

### 🔴 难点2：`map` 和 `flatMap` 有什么区别？

```java
// map：把 Optional<T> 变成 Optional<U>
// 前提：你的转换方法返回的是普通值，不是 Optional
Optional<User> userOpt = findById(1L);
Optional<String> nameOpt = userOpt.map(User::getName);      // getName() 返回 String ✅
Optional<Address> addrOpt = userOpt.map(User::getAddress);  // getAddress() 返回 Address ✅

// flatMap：把 Optional<T> 变成 Optional<U>
// 前提：你的转换方法返回的是 Optional
Optional<Address> addrOpt2 = userOpt.map(User::getAddress);  // ❌ 如果getAddress返回Optional<Address>
Optional<Address> addrOpt3 = userOpt.flatMap(User::getAddress); // ✅
```

> 🧩 **前端类比**：
> - `map` ≈ 你的函数返回普通值（`T → U`）
> - `flatMap` ≈ 你的函数返回Optional（`T → Optional<U>`）
>
> 就像JS里 `arr.map(x => [x, x*2])` 会得到二维数组，需要 `.flat()` 一下。`flatMap` = `map` + `flat`。

```typescript
// 前端对比
const optional = someFunction();  // T | null

// map 风格：返回普通值
const name = optional?.name;  // 得到 string | undefined

// flatMap 风格：返回另一个可选值
const city = optional?.address?.city;  // 连续解套
// Java里 address 返回 Optional<Address>，所以要用 flatMap
```

### 🔴 难点3：`orElse` vs `orElseGet` 的区别

```java
// orElse：不管用不用，默认值都会先计算好
String result1 = optional.orElse(computeExpensiveDefault());  // computeExpensiveDefault() 先执行了！

// orElseGet：只有optional是empty时才计算默认值（延迟计算）
String result2 = optional.orElseGet(() -> computeExpensiveDefault());  // 只有需要时才执行
```

> 🧩 **前端类比**：
> - `orElse(x)` ≈ `?? x`，但 x 总是先求值
> - `orElseGet(() => x)` ≈ 惰性求值的 `?? x`

```java
// 示例：看出区别了吗？
Optional<String> opt = Optional.of("有值！");

// orElse：即使有值，也会执行 expensiveOperation()
String a = opt.orElse(expensiveOperation());  // 控制台打印了"执行了耗时操作..."

// orElseGet：有值就不执行
String b = opt.orElseGet(() -> expensiveOperation());  // 控制台没打印
```

### 🟡 避坑：Optional 不要用在字段和参数上！

```java
// ❌ 错误用法
public class User {
    private Optional<String> email;  // 不要！Optional不支持序列化
}

public void process(Optional<String> input) {  // 不要！参数不该用Optional
    // ...
}

// ✅ 正确用法：Optional 只用在返回值
public Optional<User> findById(Long id) {  // ✅ 正确
    return Optional.ofNullable(db.get(id));
}
```

> 🧩 **前端类比**：就像你不会在每个字段上都写 `| null | undefined` 一样。Optional 只是方法返回值的"包装纸"，不是字段类型。

> **难度：⭐ | 核心概念：亲眼看看编译器的"暴怒"**

## 💻 完整代码

```java
// 文件名: CheckedExceptionExperiment.java
import java.io.*;

/**
 * 练习4：故意不catch检查型异常
 * 
 * 目的：让你亲眼看到编译器的"愤怒报错"

public class CheckedExceptionExperiment {
    
    // 这是一个会抛检查型异常的方法
    public static void throwCheckedException() throws IOException {

    }
    
    // ❌ 错误版本1：调用了 throws 方法，但自己不处理也不声明

        // 报错：Error: 未处理的异常类型 IOException
    }
    
    // ✅ 正确版本1：try-catch 处理
    public static void correctMethod1() {
        try {
            throwCheckedException();
        } catch (IOException e) {
            System.out.println("捕获到了: " + e.getMessage());
        }
    }
    
    // ✅ 正确版本2：继续往上抛 throws
    public static void correctMethod2() throws IOException {
        throwCheckedException();
    }
    
    public static void main(String[] args) {
        System.out.println("===== 练习4：故意不catch检查型异常 =====");
        
        // 调正确的版本1
        correctMethod1();
        
        // 调正确的版本2——需要在main签名上加throws
        // 你试一下：去掉下面的注释，同时在main方法签名上加 throws IOException
        // correctMethod2();
        
        System.out.println("✅ 程序正常结束");
    }
}
```

然后编译运行，看看程序怎么崩溃的：

# 🧠 Day 4 最容易翻车的4个坑（千万注意！）

### 🚨 坑1：`throws` 写在方法签名上，但以为它在"抛出"异常

```java
// ❌ 错误理解
public void doSomething() throws IOException {
    // "throws IOException" ≠ "这里抛异常"
    // 它是"声明"——告诉调用方："我这个方法可能会抛IOException"
}

// ✅ 真正抛异常的是 throw（没有s）
public void doSomething() throws IOException {
    if (somethingWrong) {
        throw new IOException("出错了");  // 👈 throw（没有s）才是真正抛出
    }
}
```

**记忆口诀**：
- `throw`（没s）= 实际抛出异常 ≈ JS的 `throw`
- `throws`（有s）= 声明可能抛异常 ≈ JS没有的概念

### 🚨 坑2：`Optional.of(null)` 直接崩

```java
// ❌ 极度危险！
Optional<User> opt = Optional.of(userMap.get(id));  // 如果 get 返回 null → NullPointerException！

// ✅ 安全做法
Optional<User> opt = Optional.ofNullable(userMap.get(id));  // 返回null就变empty
```

### 🚨 坑3：用 `Optional.get()` 而不检查

```java
Optional<User> opt = findById(999L);

// ❌ 危险！如果opt是empty，get()会抛 NoSuchElementException
User user = opt.get();

// ✅ 安全做法
User user = opt.orElseThrow(() -> new RuntimeException("用户不存在"));
// 或者
opt.ifPresent(u -> System.out.println(u.getName()));
```

> 🧩 **前端类比**：`Optional.get()` ≈ 直接访问可能为null的属性，不检查。**违反了Optional的设计初衷**！

### 🚨 坑4：检查型异常不处理就往下抛，越抛越多

```java
// ❌ 灾难——每层都 throws，调用链全是 throws
public void methodA() throws IOException { methodB(); }
public void methodB() throws IOException { methodC(); }
public void methodC() throws IOException { readFile(); }

// ✅ 正确——在最合适的地方统一处理
public void methodA() {
    try {
        methodB();
    } catch (IOException e) {
        // 在这里统一处理，或者包装成RuntimeException
        throw new RuntimeException("系统错误", e);
    }
}
public void methodB() { methodC(); }
public void methodC() { readFile(); }
```

# 💡 Day 4 定心丸——前端转Java的核心感悟

> **今天学的东西，是你从"前端自由派"转向"Java防御式编程"的关键一步。**

前端时代你习惯了：
- 到处 `?.` 和 `??` 优雅处理空值
- JSON.parse 崩了就崩了，catch不catch随你
- 对象属性随时可能 undefined，你早就麻木了

但Java告诉你：
1. **所有代码都可能有异常**，但你要么处理，要么声明——不能假装它不存在
2. **null 是十亿美金错误**，用 Optional 强迫你正视它
3. **编译器是你的盟友**，你改代码时它帮你检查遗漏

**这种感觉一开始会很不舒服**——"干嘛管这么多？我写了20年前端从来没处理过这些！"

但几个月后你会发现，**Java写的代码，尤其是在团队协作中，bug少得惊人**。这就是"防御式编程"的威力——**把错误消灭在编译时，而不是运行时**。
